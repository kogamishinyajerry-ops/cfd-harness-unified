## Open follow-up (not blocking M5.0 ship)

**Mock-solver run E2E on imported case**: spec_v2 deliverable but not
verified in this commit. Imported cases get
`flow_type: imported_unspecified` + `geometry_type: imported_stl` (per
`manifest_writer._editor_case_yaml`); the existing `wizard_drivers`
solver-driver dispatch is keyed on case_id matching the whitelist. An
imported case has no whitelist entry, so wizard SSE will not produce a
real-solver invocation today.

Two paths to close this gap:

1. **Quick patch** (~30 LOC, M5.0.1 follow-up): teach
   `wizard_drivers.MockSolverDriver` to recognize imported cases via the
   `source_origin == "imported_user"` field and emit a synthetic mock
   run; gates the "Run with these params" button in `EditCasePage`
   on having an OpenFOAM-runnable case dir, which imported cases do
   not yet (sHM stub is stubbed-by-design until M7).
2. **Right-shape**: defer the run capability to M7 entirely. M5.0 ships
   "import → editor", and the user iterates parameters in the editor
   without running until M7 lands snappyHexMesh + real-solver dispatch
   for imported geometries.

Choosing **path 2** for ship is consistent with the spec_v2 tier split
(M5.0 ≠ M5.1 ≠ M7). The mock-solver E2E requirement in the original v1
kickoff was authored before the tier split; spec_v2's M5.0 surface does
not include a runnable case path.

This deviation is recorded for review.
