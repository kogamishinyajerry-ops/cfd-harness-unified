## Why M-PANELS now

CFDJerry's Pivot Charter Addendum 3 §1 trigger quote (the ANSYS-class
workbench requirement) drove three milestones in sequence:

- M-VIZ shipped the viewport (proves rendering works).
- M-RENDER-API shipped the data feed (proves the viewport can show
  geometry / mesh / fields from a real case).
- **M-PANELS now wires both into the engineer-driven step-panel UI**
  the charter §3 hard constraint promises:

> "工程师可以像操作 ANSYS Fluent 一样，控制每个操作细节…可以点击
> 'AI 处理'，执行自动化设置、操作、让主区域内容更新，但是不自动跳转
> 下一步，而是要工程师点击'下一步'。"

Without M-PANELS, the viewport (M-VIZ) and the render endpoints
(M-RENDER-API) have no consumer in the workbench UI — they exist as
disconnected primitives. ImportPage / EditCasePage / MeshWizardPage /
WizardRunPage today are discrete SSE-wizard routes with no 3D viewport,
no step-tree, no per-step AI buttons. That is the agentic-wizard
paradigm Addendum 3 explicitly retired.

M-PANELS replaces that paradigm with the engineer-driven three-pane
workbench shell:

```
┌─────────────────────────────────────────────────────────────────────┐
│ Top bar · case identity · global actions · save indicator           │
├──────────┬──────────────────────────────────────────────┬───────────┤
│ Left     │  Center                                      │ Right     │
│ ──────   │  ──────                                      │ ──────    │
│ 5-step   │  3D viewport (M-VIZ Viewport · format='glb') │ Task      │
│ tree     │  consuming M-RENDER-API endpoints based on   │ panel     │
│          │  the active step                             │ for the   │
│ Step 1   │                                              │ active    │
│  Import  │  Step 1 → /geometry/render (M-RENDER-API)    │ step      │
│ Step 2   │  Step 2 → /mesh/render     (M-RENDER-API)    │           │
│  Mesh    │  Step 3 → boundary patch overlay (Tier-B)    │ [AI 处理] │
│ Step 3   │  Step 4 → live residuals + log (Tier-B)      │ [上一步]  │
│  Setup   │  Step 5 → /field/{name}    (M-RENDER-API)    │ [下一步]  │
│ Step 4   │                                              │           │
│  Solve   │                                              │ form      │
│ Step 5   │                                              │ inputs +  │
│  Results │                                              │ status    │
├──────────┴──────────────────────────────────────────────┴───────────┤
│ Bottom strip · log tail · last action · validation status           │
└─────────────────────────────────────────────────────────────────────┘
```

This is the canonical CAE-tool layout (ANSYS Fluent Workbench, Star-CCM+,
ParaView). The §3 product-interaction hard constraint is fundamentally
**a layout + interaction-mode commitment**; M-PANELS is where it lands.

## Goal (one sentence)

**Land the three-pane workbench shell — left step-tree (5 steps · Import,
Mesh, Setup, Solve, Results), center 3D viewport consuming M-RENDER-API
endpoints based on the active step, right task-panel with the
`[AI 处理] / [上一步] / [下一步]` button contract, bottom status strip
— Tier-A wires Steps 1 (Import) and 2 (Mesh) to existing flows
end-to-end, with the remaining steps showing placeholder task-panels
that get filled in by M-AI-COPILOT and M7-redefined.**

## Scope tier split

### Tier A · MVP (this milestone must ship)

1. **Three-pane workbench layout** under
   `ui/frontend/src/pages/workbench/StepPanelShell.tsx`:
   - Top bar (case identity + save indicator + close button)
   - Left rail: `<StepTree>` rendering the 5 steps with active /
     completed / pending visual states
   - Center: `<Viewport format='glb' glbUrl=...>` (from M-VIZ) wired
     to the active step's M-RENDER-API endpoint
   - Right rail: `<TaskPanel>` rendering the active step's form +
     `[AI 处理] / [上一步] / [下一步]` button contract
   - Bottom strip: `<StatusStrip>` showing log tail + last-action +
     validation status

2. **Step-tree component (`<StepTree>`)** as a controlled component:
   - Props: `currentStepId`, `stepStates` (per-step status: pending /
     active / completed / error), `onStepClick(stepId)`
   - Visual states match Fluent / Star-CCM+ convention (active =
     accent color, completed = check, error = red dot, pending =
     muted)
   - Click jumps to the step (with confirmation if the current step
     has unsaved changes)

3. **Three-button contract (`[AI 处理] / [上一步] / [下一步]`)**:
   - `[AI 处理]` is per-step opt-in automation. M-PANELS Tier-A
     **declares the contract** (button + endpoint shape) but
     individual step wire-ups land in M-AI-COPILOT.
   - `[上一步] / [下一步]` are explicit navigation — no auto-advance
     on AI 处理 success (Addendum 3 §3.b binding).
   - Disabled state: when the active step has unmet validation
     prerequisites, `[下一步]` is disabled with a tooltip explaining
     why.

4. **Step 1 (Import) wired up**:
   - Reuses existing `ImportPage` upload flow + STL preview
   - **Switches Viewport from `format='stl'` to `format='glb'`** (uses
     M-RENDER-API's `/geometry/render` endpoint)
   - On successful import, marks Step 1 completed; user clicks
     `[下一步]` to advance.

5. **Step 2 (Mesh) wired up**:
   - Task-panel surfaces the existing mesh-wizard form (mesh size,
     refinement region, geometry-aware defaults from gmsh)
   - Center viewport shows the polyMesh wireframe via M-RENDER-API's
     `/mesh/render` endpoint
   - `[AI 处理]` placeholder fires the existing `/api/cases/<id>/mesh`
     endpoint that triggers gmsh + sHM (M6.0 territory)
   - On success, marks Step 2 completed.

6. **Steps 3-5 (Setup / Solve / Results) placeholder task-panels**:
   - Step 3: `<SetupPanelPlaceholder>` — shows "Setup wires up in
     M-AI-COPILOT" + a link to the live `case_manifest.yaml` (read-
     only preview) + the disabled-by-design `[AI 处理]` button.
   - Step 4: `<SolvePanelPlaceholder>` — shows the existing
     `WizardRunPage` SSE log behind a static panel.
   - Step 5: `<ResultsPanelPlaceholder>` — shows the existing
     `BcCheckCard` + completion artifacts list, with a stub for the
     field overlay (M-VIZ.results).

7. **Routing**:
   - New route `/workbench/<case_id>` lands the StepPanelShell.
   - Existing routes (`/workbench/<case_id>/import`,
     `/workbench/<case_id>/edit`, `/workbench/<case_id>/mesh`,
     `/workbench/<case_id>/run`) get redirects to the new shell with
     `?step=N` query param so deep links remain stable for one
     milestone (then deprecated in M7-redefined).

8. **Backend (zero new endpoints in Tier-A)**:
   - Reuses M-RENDER-API endpoints for the center pane.
   - Reuses existing `/api/cases/<id>/manifest` / `/api/cases/<id>/mesh`
     / `/api/cases/<id>/runs` endpoints for the task-panel forms.
   - **The face-naming patch endpoint that the deferred doc proposed
     stays deferred to Tier-B / M-PANELS.advanced.**

9. **§11.1 BREAK_FREEZE**: M-PANELS Tier-A WILL touch
   `ui/frontend/src/pages/workbench/**` (this is the whole point).
   The kickoff DEC declares `BREAK_FREEZE: rationale=Addendum 3 §3
   product-interaction hard constraint binding` per Charter §5
   选项一 (per-PR escape clause). Counter increments by 1 on the
   first M-PANELS implementation commit (M-VIZ used 1 of 3; M-PANELS
   uses 1 of remaining 2; M-AI-COPILOT will likely use 1 of 1).
   See `spec_v2 §BREAK_FREEZE accounting` for full ledger.

## Library + dependency choice

Zero new external dependencies needed for Tier-A:
- **React + React Router** (already in the stack · used by current
  workbench pages)
- **Tailwind** (already in the stack · used for ImportPage / current
  workbench)
- **`@kitware/vtk.js`** (added in M-VIZ · already lazy-chunked)
- **`@tanstack/react-query`** for endpoint state — *already used by
  several existing pages* (verify ImportPage's actual data hook
  during the surface scan; if absent, defer the query refactor)

Specifically *not* added:
- **No new UI component library** — Tailwind primitives + a thin
  `<Pane>` / `<Button>` / `<FormRow>` wrapper layer is enough for
  Tier-A. Material UI / Chakra / shadcn would be over-budget.
- **No state-management library** (no Redux / Zustand / Jotai). Step
  state lives in the URL (`?step=N`) + react-query for server state.
  The 5-step shell does not need cross-tree state.

## Library / framework risk

- **vtk.js Viewport already chunk-split** — M-PANELS adds the SHELL
  around it; bundle delta should be modest (mostly Tailwind class
  expansion + one new layout component tree). spec_v2 §AC#10
  carries an explicit +50 KB gz delta budget.
- **react-query usage may need to be added if the current workbench
  pages don't use it** — surface scan during Step 1 of spec_v2
  §Sequence verifies; if a refactor is needed, it is line-A bounded.

## Self-pass-rate estimate

**60%** — calibrated -15% vs M-RENDER-API (75%) and -10% vs M-VIZ (70%)
because:
- **Largest UX surface yet** — three-pane shell is structurally
  novel; every prior milestone reused page-level patterns.
- **§11.1 BREAK_FREEZE consumed** — every PR in this arc takes a slot
  out of the remaining 2/3 quota.
- **Routing redirect contract is tricky** — deep links from prior
  workbench pages have to keep working for one milestone, then
  deprecate cleanly in M7.
- **vtk.js + react-query + form-state interaction** has multi-axis
  failure modes (state sync, abort signals, stale glb chunks across
  step navigation).

Per RETRO-V61-001: ≤70% self-pass triggers **pre-merge Codex review**
(not post-merge). M-PANELS pre-merge Codex is mandatory.
