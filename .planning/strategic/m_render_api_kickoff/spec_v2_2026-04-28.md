## Library choice — trimesh.export (already in `[workbench]` extra)

### Tradeoff table

| Library | Bundle / runtime cost | STL → glTF | polyMesh wireframe | Field encoding | Verdict |
|---|---|---|---|---|---|
| **trimesh.export** | already a [workbench] dep · 0 KB delta | ✅ native (`.export(file_type='glb')` returns binary glb · verified 2026-04-28: 12-tri cube → 960 bytes glb) | ❌ no native polyMesh — but we don't need it; we extract edges manually from points/faces files, then trimesh assembles the line glTF | ⚠️ no native scalar-field support — we hand-pack accessor for Tier A | **Working assumption** |
| **pygltflib** | new dep · ~50 KB · no STL parse | ❌ (gltf assembly only) | ✅ would assemble line glTF from raw points/indices | ✅ scalar-field accessor packing well-documented | Not chosen for geometry; **considered for field endpoint** if trimesh accessor ergonomics are insufficient |
| **vtkpython** (CPython binding) | huge runtime install + ABI risk | ✅ via vtk.js's source format | ✅ native polyMesh reader | ✅ native field reader | Rejected: dep-weight >> M-RENDER-API value |
| **OpenFOAM `foamToVTK`** + glTF transcode | requires OpenFOAM container · slow | ✅ | ✅ | ✅ | Rejected: container round-trip too slow for interactive use |

**Decision rationale**:

1. trimesh covers Tier-A geometry endpoint with **zero new dependencies** —
   the heaviest M-RENDER-API risk class (new dep introduction) is avoided.
2. polyMesh wireframe extraction is a small parser + trimesh line-glTF
   assembly. ~250 LOC. No new lib needed.
3. Field-sample is the trickiest: we hand-pack a glTF accessor with
   binary float32 buffer. If trimesh's primitive API can't express it,
   we fall back to **pygltflib** (small dep, pure-Python, no native
   build chain) or to a non-glTF binary stream channel for Tier-A
   stub. Decision deferred to Step 4.
4. M-RENDER-API migration cost (changing libraries mid-project) is
   moderate — picking trimesh now and committing avoids the
   "first I'll try foo, then realize bar is better, then port" loop.

**Override window**: CFDJerry can ratify a different library in the
DEC `library_choice` field before flipping Status to Accepted. Past
that, library change requires a new DEC.

## Tier A · MVP scope (this milestone ships)

### Backend deliverables

#### B.1 Geometry render endpoint

```
GET /api/cases/<case_id>/geometry/render
  → 200 + binary glb + Content-Type: model/gltf-binary
  → 404 if case_id not found
  → 422 if source format unsupported (no .stl on disk)
  → 503 if [workbench] extra not installed (mirrors M5.0 import_geometry)
```

**Implementation** at `ui/backend/routes/geometry_render.py` (NEW
endpoint added to existing module from M-VIZ Step 3):

```python
@router.get("/cases/{case_id}/geometry/render", tags=["geometry-render"])
def get_case_geometry_glb(case_id: str) -> Response:
    # 1. Validate case_id (is_safe_case_id)
    # 2. Locate source STL: same path as /geometry/stl
    # 3. Check cache: <case_dir>/.render_cache/geometry.glb mtime
    # 4. If cache miss: trimesh.load(stl_path).export(file_type='glb')
    #    → write to cache atomically (tempfile + os.replace)
    # 5. Return FileResponse(cache_path, media_type='model/gltf-binary')
```

~80 LOC. Cache invalidation by **source-mtime comparison** (cache
mtime ≥ source mtime → hit; otherwise miss). Atomic-rename pattern
on write to prevent partial-cache reads under concurrent requests.

#### B.2 Mesh wireframe endpoint

```
GET /api/cases/<case_id>/mesh/render
  → 200 + binary glb (LINES primitive) + Content-Type: model/gltf-binary
  → 404 if case_id not found OR no polyMesh on disk
  → 422 if polyMesh malformed
```

**Implementation** at `ui/backend/services/render/mesh_wireframe.py`
(NEW module under anticipatory line-A path declared in `8fdb0a3`):

1. Locate `<case_dir>/constant/polyMesh/` (M6.0 output) — fall back
   to user's `imported_case_dir/constant/polyMesh/` per template_clone
   convention.
2. Parse `points` (vertex array) and `faces` (face → vertex indices) —
   OpenFOAM ASCII or binary; trimesh has helpers.
3. Extract unique edges from face-edge connectivity. Use a sorted-pair
   set for deduplication; ~3-5× face_count typical edges.
4. Assemble glTF `mode: LINES` primitive with the extracted edges as
   `indices` accessor over the original points buffer.
5. Cache to `<case_dir>/.render_cache/mesh.glb` keyed on `points` +
   `faces` mtimes (the two files that change when sHM re-runs).

~150-250 LOC including the polyMesh parser; ~50 LOC if we delegate
parsing to a future foam_io helper.

**Tier-A simplification**: only `mode: LINES` primitive (cell edges).
NOT `mode: TRIANGLES` (mesh surface) and NOT `mode: POINTS` (cell
centers) — those are M-VIZ.mesh follow-ups.

#### B.3 Field sample endpoint stub

```
GET /api/cases/<case_id>/results/<run_id>/field/<name>
  → 200 + binary glb (TRIANGLES with COLOR_0 accessor) + Content-Type: model/gltf-binary
       OR
  → 200 + application/octet-stream binary float32 array (Tier-A fallback)
  → 404 if case_id / run_id / field name not found
  → 422 if field type unsupported (vector / tensor in Tier A)
```

**Implementation** at `ui/backend/services/render/field_sample.py`
(NEW · anticipatory path · ~80 LOC):

Tier-A scope:
- **Scalar fields only** (pressure `p`, velocity magnitude `Umag`,
  temperature `T`, turbulence quantities `k` / `omega` / `epsilon`)
- **internalField only** (no boundary fields in Tier A)
- Source: OpenFOAM time-directory at `<case_dir>/<time>/<name>` per
  the run_history convention; or `reports/phase5_fields/<case>/<ts>/`
  for completed runs (per comparison_report.py:40).
- Output: glTF binary glb with the geometry-render polydata
  augmented by a `COLOR_0` accessor packed from the scalar values
  (RGB linearly mapped from min→max with default colormap).

**Tier-A fallback**: if glTF accessor packing in trimesh proves
ergonomically awkward (likely; trimesh's COLOR_0 surface is
underdocumented), Tier-A SHIPS as `application/octet-stream` binary
float32 array — frontend handles colormap mapping. M-VIZ.results
upgrades to full glTF with baked colors.

#### B.4 Cache invalidation + cleanup

```
<case_dir>/.render_cache/
  ├── geometry.glb   (keyed on triSurface/*.stl mtimes)
  ├── mesh.glb       (keyed on constant/polyMesh/{points,faces} mtimes)
  └── field-<run_id>-<name>.glb    (keyed on <run_id>/<time>/<name> mtime)
```

Never cleaned automatically in Tier-A (if disk pressure becomes a
concern, M-RENDER-API.perf adds LRU eviction). Invalidation = source-
mtime comparison only; "stale" cache files of deleted runs persist
harmlessly until manual cleanup or `.render_cache/` is wiped.

### Frontend deliverables

#### F.1 Viewport `format` prop

`ui/frontend/src/visualization/Viewport.tsx` gains an optional `format`
prop:

```tsx
interface ViewportProps {
  stlUrl: string;                       // legacy STL URL (M-VIZ)
  format?: 'stl' | 'glb';              // NEW · default 'stl'
  glbUrl?: string;                      // NEW · used when format='glb'
  width?: number;
  height?: number;
  background?: [number, number, number];
}
```

When `format='glb'`:
- `stl_loader.ts` is bypassed; instead `glb_loader.ts` (NEW · ~50 LOC)
  uses `vtkGLTFImporter` to load the glb URL directly into the renderer.
- The kernel's `attachStl` becomes `attachGltf`.

When `format` omitted → behavior identical to M-VIZ Tier-A. **No
regression** to the M-VIZ smoke-tested flow.

ImportPage NOT touched in Tier-A — it still uses STL via the existing
`stlUrl` path. M-PANELS will introduce the `format='glb'` consumer.

#### F.2 `glb_loader.ts` (NEW)

Mirrors `stl_loader.ts` shape:
- `fetchGlbBytes(url, signal): Promise<ArrayBuffer>` — same fetch
  semantics + same StlLoadError → renamed `RenderLoadError` with
  shared `kind: 'fetch' | 'parse'` enum.
- `parseGlbBytes(buffer): GlbData` — uses `vtkGLTFImporter` (new
  selective import).
- `loadGlbFromUrl(url, signal): Promise<GlbData>` — composes.

vtk.js side: `viewport_kernel.ts::createKernel` adds an `attachGltf`
method analogous to `attachStl` but using `vtkGLTFImporter`'s scene
output (which already provides actors with mappers). Disposal order:
the importer's actors → reader → grw → interactor (mirrors STL).

#### F.3 No ImportPage edit in Tier-A

ImportPage.tsx unchanged. The `format='glb'` integration lands in
M-PANELS Step 2 (or when M-VIZ.mesh / M-VIZ.results land — whichever
comes first).

## Performance budget (Tier A)

- Geometry-render cache miss (3 bundled fixtures): ≤ 200 ms server-side
  (trimesh STL parse + glb export · benchmarked separately)
- Geometry-render cache hit: ≤ 20 ms (FileResponse on cached glb)
- Mesh-render: ≤ 1 s on a 5k-cell polyMesh
- Field-sample: ≤ 500 ms on a 100k-point internalField
- Frontend glb load + render (warm cache): ≤ 800 ms on M3 Pro

If any budget violated → P0 / blocker.

## Risk register

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| trimesh glb output non-conformant on edge-case STL (multi-solid, weird units) | Low | Medium | Smoke test against the 3 fixtures + 1 multi-solid ASCII STL; document any quirks |
| polyMesh edge-extraction performance on >100k cells | Low (bundled fixtures all <1k cells) | Low | Defer to M-RENDER-API.perf |
| glTF accessor packing for COLOR_0 too awkward in trimesh | Medium | Low | Tier-A fallback to `application/octet-stream` binary float32; M-VIZ.results upgrades |
| vtk.js `vtkGLTFImporter` divergence from server-side glTF | Low | Medium | Unit test against round-tripped fixture (encode then decode then compare attribute counts) |
| Cache directory hits permission issues on user_drafts/ | Low | Low | mkdir(parents=True, exist_ok=True) + same FS permissions as parent |
| Concurrent writes to cache file | Low | Medium | atomic-rename pattern (tempfile.NamedTemporaryFile + os.replace) |
| Frontend `glb_loader` shares too much code with `stl_loader` (refactor opportunity? premature abstraction?) | Medium | Low | Keep them separate in Tier-A; revisit when third loader (mesh? field?) lands and pattern is clear |
| §11.1 freeze pressure | Low | Low | M-RENDER-API touches NEW paths only (`ui/backend/routes/geometry_render.py` extension already exists; new backend services under `ui/backend/services/render/**` are line-A NEW; `ui/frontend/src/visualization/**` was already line-A NEW). BREAK_FREEZE count remains 1 from M-VIZ. |
| Bundle bloat from vtkGLTFImporter | Medium | Medium | Selective import (`@kitware/vtk.js/IO/Geometry/GLTFReader` or equivalent) + measure delta in Step 4 |

- M-VIZ first-pass effective ~50% (6 rounds of fixes for 4 P2 + 1 P1 + 2 P3 findings)

## Path-A Gate 2 demo flow

Gate-1 already unlocked (M-VIZ MVP merged 2026-04-28 commit `7642bb4`).

Gate 2 unlocks at M-RENDER-API merge: stranger uploads their geometry,
runs M6.0 mesh generation (manual via existing API), and the Viewport
displays the **mesh wireframe** on top of the geometry. First "you can
see the mesh" moment.

Demo script:
1. Stranger has uploaded their STL (Gate-1 already worked)
2. Engineer (or stranger via UI) triggers M6.0 mesh generation
3. Frontend page (still ImportPage in M-RENDER-API; M-PANELS later)
   adds a "Show mesh" toggle that swaps the Viewport's URL from
   `/api/cases/<id>/geometry/stl` → `/api/cases/<id>/mesh/render`
4. Viewport re-renders showing cell edges
5. Stranger sees the mesh — second "this is real" moment

If no stranger by Gate-2: internal demo, recruitment continues in
parallel.
