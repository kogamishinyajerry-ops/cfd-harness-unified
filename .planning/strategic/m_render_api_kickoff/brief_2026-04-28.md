## Why M-RENDER-API now

M-VIZ Tier-A shipped a single anticipatory STL-serve endpoint (`GET
/api/cases/<id>/geometry/stl`). That is intentionally minimal — just
enough to feed the Viewport with imported-case geometry. M-RENDER-API
expands the surface to the full set of render-data endpoints the
forthcoming step-panel UI needs:

- **Geometry**: not just the original imported STL — also generated
  geometry from the gmsh / sHM pipeline (M6.0 territory) and arbitrary
  CAD formats (future). Browser-friendly format = **glTF binary**
  (.glb), transcoded server-side from whichever native format the
  pipeline emitted.
- **Mesh**: polyMesh wireframe (line geometry) so engineers can see the
  cell structure on the imported geometry. M-PANELS step 3 ("Mesh")
  needs this — without it, the Mesh step has no observable surface.
- **Field**: sampled scalar fields (pressure, velocity magnitude,
  k, omega, T, ...) on the meshed geometry, after a solver run
  completes. M-PANELS step 5 ("Results") needs this. Field rendering
  is M-VIZ.results scope on the frontend; the backend endpoint stub
  lands in M-RENDER-API so the contract is defined early.

Per Pivot Charter Addendum 3 §4.c hard ordering: M-VIZ proves the
viewport works → M-RENDER-API expands the data feed → M-PANELS wires
the three-pane layout that consumes it.

## Goal (one sentence)

**Land three backend endpoints — `GET /api/cases/<id>/geometry/render`
(glTF), `GET /api/cases/<id>/mesh/render` (polyMesh wireframe glTF),
and `GET /api/cases/<id>/results/<run>/field/<name>` (sampled scalar
field) — under the existing line-A render-services tree
(`ui/backend/services/render/**` declared in commit `8fdb0a3`),
smoke-tested end-to-end against the bundled fixtures + at least one
M6.0-meshed imported case + at least one M5.1-completed run.**

## Scope tier split

### Tier A · MVP (this milestone must ship)

1. **Geometry render endpoint** at `GET /api/cases/<id>/geometry/render`:
   - Reads STL or any pipeline-produced source under
     `user_drafts/imported/<case_id>/triSurface/` or
     `user_drafts/<case_id>/constant/triSurface/` (M6.0-meshed)
   - Transcodes to **binary glTF (.glb)** server-side
   - Returns `Content-Type: model/gltf-binary` + `Content-Disposition: attachment`
   - 404 if case not found · 422 if source format unsupported
   - Library: **trimesh.export** (already a [workbench] dep · supports
     STL → glTF natively) — no new dependencies needed for Tier A
   - Caches transcoded glb to `<case_dir>/.render_cache/geometry.glb`
     keyed by source-file mtime to skip repeat work

2. **Mesh wireframe endpoint** at `GET /api/cases/<id>/mesh/render`:
   - Reads `<case_dir>/constant/polyMesh/` (M6.0 output)
   - Extracts cell edges as line geometry
   - Emits glTF with `mode: LINES` primitives
   - 404 if no polyMesh on disk · returns the geometry-render glTF
     fallback if mesh is decimal-truncated polyMesh-on-stl (M6.0 modes)
   - Implementation strategy: parse polyMesh `points` + `faces` files
     directly (no gmsh round-trip needed); cell edges extracted via
     face-edge connectivity. ~150-250 LOC.

3. **Field sample endpoint stub** at `GET /api/cases/<id>/results/<run>/field/<name>`:
   - Reads OpenFOAM time-directory (`<case_dir>/<time>/<name>`)
   - Returns sampled scalar field as binary float32 array packed in
     glTF accessor (or a separate `application/octet-stream` channel
     if glTF accessor encoding is too involved for Tier A)
   - 404 if run not found · 422 if field name not in run
   - Tier A may ship as **stub-with-real-payload-format** (returns
     correct shape but only on a single hand-prepared fixture); full
     coverage = M-VIZ.results scope

4. **Backend tests** (pytest):
   - geometry_render: ≥3 fixtures (the bundled M-VIZ fixtures) + STL
     → glTF transcoding round-trip via trimesh
   - mesh_render: 1 M6.0-meshed case (e.g. ldc_box after gmsh run)
   - field_sample: 1 M5.1-completed run (any case where a solver ran
     to completion)

5. **Frontend smoke-only** (no Viewport changes yet):
   - Update Viewport to optionally fetch glTF instead of STL via a
     `format` prop; if not specified, defaults to STL (preserves M-VIZ
     behavior). Mesh wireframe rendering and field rendering are
     **NOT** wired into Viewport in Tier A — those are M-VIZ.mesh /
     M-VIZ.results scope.

### Tier B · deferred (NOT this milestone)

- Multi-file STL transcoding (combine `triSurface/inlet.stl + outlet.stl`
  → single glTF) → M-RENDER-API.advanced
- ASCII OpenFOAM → binary glTF for fields (need to write a custom
  packer; ASCII parsing is heavy) → M-VIZ.results
- Vector field (velocity glyphs) → M-VIZ.results
- Streamline integration → M-VIZ.results
- Server-side decimation for >1M-face meshes → M-RENDER-API.perf
- Texture / colormap baked into glTF → M-VIZ.results

## Self-pass-rate estimate

**75%** — calibrated against M-VIZ (~50% effective first-pass after
6 rounds of fixes) + M5.1 (~70% first-pass with 1 Codex round).
M-RENDER-API has lower integration risk than M-VIZ:

- All backend-only changes (no WebGL lifecycle class)
- trimesh is already battle-tested in the codebase (M5.0 ingest)
- glTF format is well-specified (no library-divergence surprises)
- Frontend touch is minimal (one optional prop)

## Top failure modes anticipated

1. **trimesh.export glTF emits non-conformant binary glb on edge cases
   (multi-solid STL, units != m)** → fallback to ASCII glTF + manual
   embedding of binary buffer; document in test
2. **polyMesh edge extraction performance** on large fixtures (>100k
   cells) → defer to M-RENDER-API.perf · current fixtures all <10k
3. **OpenFOAM ASCII field parser** corner cases (boundary fields,
   multi-zone) → Tier A scope is internalField only; boundary field
   parsing deferred
4. **glb cache invalidation race** on concurrent reads + writes →
   atomic-rename pattern · test
5. **Browser glTF compatibility** with vtk.js's glTF loader (vs three.js)
   → smoke against vtk.js's vtkGLTFImporter; document any vtk.js-specific
   quirks
