## Library choice — vtk.js (working assumption · subject to ratification)

### Tradeoff table

| Library | Bundle (gzipped) | CAE primitives | Learning curve | TypeScript | Long-term fit | Verdict |
|---|---|---|---|---|---|---|
| **vtk.js** (Kitware) | ~600 KB core + ~400 KB STL/glTF + ~2 MB total uncompressed | ✅ Native (contour / streamline / glyph / clip / cut all in-box) | Medium-high (VTK pipeline mental model) | Partial (auto-generated from JSDoc; some `any` accepted) | ✅ Step 5 Results lands naturally | **Working assumption** |
| **three.js** | ~150 KB core | ❌ Not built for scientific viz; contours/streamlines need handrolling | Low-medium | ✅ First-class | ❌ Hits wall at Step 5 | Not chosen |
| **paraview-glance** (Kitware vtk.js wrapper) | ~3 MB | ✅ ParaView-pipeline ready | Low (declarative) | ⚠️ Less ergonomic for embedding our `[AI 处理]` button per step | ⚠️ Harder to customize step-panel integration | Not chosen |
| **regl** (functional WebGL) | ~50 KB | ❌ Bare-metal, no CAE | Low | ✅ | ❌ | Not chosen |
| **babylon.js** | ~700 KB | ❌ Game engine focus | Low-medium | ✅ | ❌ | Not chosen |

**Decision rationale**:

1. CAE primitives matter for downstream M-VIZ.results (contours, streamlines). Three.js requires custom shaders + ad-hoc geometry generation for those — that's months of work we'd otherwise inherit free from vtk.js.
2. Bundle cost (2 MB raw / ~1 MB gzipped) is acceptable for a CAE workbench. Engineers running CFD don't browse from 3G.
3. paraview-glance is the right answer if we wanted to drop in a wholesale viewer — but it doesn't compose with per-step `[AI 处理]` buttons cleanly. We need vtk.js as a library, not paraview-glance as an app shell.
4. M-VIZ migration cost (changing libraries mid-project) is high — picking vtk.js now and committing avoids the regret-and-port scenario most projects hit.

**Override window**: CFDJerry can ratify a different library in DEC-V61-094 by editing the `library_choice` field in DEC frontmatter before flipping Status to Accepted. Past that, library change requires a new DEC.


**Capabilities** (Tier A):
- Mounts vtk.js render window into container `<div>`
- Loads STL via `vtkSTLReader.parseAsArrayBuffer` from `stlUrl`
- Trackball + orbit camera (vtk.js default `vtkInteractorStyleTrackballCamera`)
- Pan + zoom (mouse wheel + middle-click drag)
- "Reset camera" exposed as imperative ref method or button (UI placement TBD in M-PANELS)
- Cleanup: `renderWindow.delete()` + `interactor.delete()` on unmount → no WebGL leak

**Out for Tier A**:
- No mesh / contour / streamline rendering (Tier B)
- No section-cut / clip-plane (Tier B)
- No screenshot export (Tier B)

### Backend deliverables

#### B.1 STL-serve endpoint

```
GET /api/cases/<case_id>/geometry/stl
  → 200 + STL binary + Content-Type: model/stl
  → 404 if case_id not in user_drafts/imported/
  → 404 if no STL on disk in <case_dir>/triSurface/
```

**Implementation**: minimal route at `ui/backend/routes/geometry_render.py`. Reads from `user_drafts/imported/<case_id>/triSurface/*.stl`, streams the file. ~30 LOC.

**NOT** the full M-RENDER-API spec — that's a later milestone with multiple formats, mesh, fields. M-VIZ ships only the STL serve as the immediate consumer for the Viewport.

## Performance budget (Tier A)

- First viewport mount + STL load (warm cache): ≤ 500 ms on a typical dev laptop (M3 Pro / equivalent)
- Camera interaction: ≥ 30 fps for our 3 fixtures
- Bundle: vtk.js + dependencies ≤ ~1 MB gzipped delta vs current bundle
- Memory: no leak observed on 100 mount/unmount cycles in dev tools

If any budget violated → P0 / blocker for milestone close.

## Risk register

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| vtk.js bundle exceeds 3 MB total | Medium | High (perf budget breach = milestone reopens) | Tree-shake imports · use `@kitware/vtk.js/Sources/...` selective imports rather than barrel imports · webpack/vite analyzer in CI |
| WebGL context leak | Medium | High (browser tab crash after N navs) | Explicit cleanup in `useEffect` return · regression test mocks render-window dispose |
| vtk.js TS types incomplete | High | Medium (some `any`) | Wrap `any` boundaries in our own typed interfaces · document accepted `any` inline |
| STL loader fails on a fixture (binary vs ASCII) | Low (all 3 are binary, valid) | Low | Fallback to ASCII parser if binary fails · clear error message with hint |
| Browser compat issue (Safari WebGL quirks) | Low | Medium | Smoke on Chrome + Safari · document Edge / Firefox as untested-but-expected-OK |
| Path A stranger recruited week 1 with no demo until week 2-3 | High | Low (engagement policy covers it) | Engagement policy: async mockup feedback in week 1 · early viewport demo at week 2-3 ship |
| Performance on a future 100k-face STL | Low (no current fixture) | Low | Documented as M-VIZ.advanced scope · current fixtures all <1k faces |
| §11.1 freeze interaction | Low | Low | M-VIZ paths (`ui/frontend/src/visualization/**`) are NEW, not in freeze scope (`ui/frontend/src/pages/workbench/**`); ImportPage tweak (A.2) IS in freeze scope → that PR cites BREAK_FREEZE referencing Addendum 3 §3.a |
