## Why this is M-VIZ first

Pivot Charter Addendum 3 §3.a makes 3D viewport the product center. Without a working viewport, every downstream milestone is a structural empty shell:

- M-RENDER-API has no consumer for its outputs → can't validate endpoint contracts
- M-PANELS shells produce a three-pane layout with the center pane blank → can't ship to user
- M-AI-COPILOT depends on viewport updates after `[AI 处理]` invocations → no observable surface

So M-VIZ is the foundational milestone. It cannot be skipped, parallelized away, or deferred.

## Goal (one sentence)

**Land a 3D viewport React component that renders STL geometry of an imported case with trackball / orbit / pan / zoom camera controls, embedded in the existing workbench routes, smoke-tested against the three bundled fixtures.**

## Scope tier split (MVP vs deferred)

### Tier A · MVP (this milestone must ship)

1. **Library selection** — vtk.js (working assumption per α direction) with explicit tradeoff table for override before ratification
2. **Viewport React component** at `ui/frontend/src/visualization/Viewport.tsx`:
   - Mounts a vtk.js render window into a `<div>` ref
   - Loads STL via `vtkSTLReader` from a backend-served URL
   - Trackball / orbit camera (pan + rotate + zoom)
   - Reset-camera button
   - Cleanup on unmount (no canvas / WebGL leaks)
3. **Backend STL-serve endpoint** at `GET /api/cases/<case_id>/geometry/stl`:
   - Reads `triSurface/<filename>.stl` from imported case dir
   - Returns binary STL with `Content-Type: model/stl` (or `application/octet-stream` if browser-render compat preferred)
   - 404 if case not imported
   - This is a tiny anticipatory endpoint — full M-RENDER-API spec (multi-format, glTF, mesh, fields) lands later
4. **Wire viewport into ImportPage** as "Geometry preview" panel (NOT replacing the report card — adding a viewport above/beside it). This is intentional NON-Fluent-style integration; M-PANELS will replace the layout properly later. M-VIZ gives the viewport a smoke-test home immediately.
5. **Smoke tests** (frontend Vitest + backend pytest):
   - Backend: `GET /api/cases/<id>/geometry/stl` returns 200 + STL bytes for each of 3 bundled fixtures (ldc_box, cylinder, naca0012)
   - Frontend: viewport component mounts, loads a fake STL, calls expected vtk.js APIs (mocked) — no actual WebGL rendering in CI

### Tier B · deferred (NOT this milestone)

- polyMesh wireframe rendering → M-VIZ.mesh follow-up after M-RENDER-API
- Contour / vector / streamline plots → M-VIZ.results follow-up
- glTF transcoding for non-STL geometry formats → M-RENDER-API
- Section-cut / clip-plane / box-clip → M-VIZ.advanced
- Multi-pane viewport (4-up, ParaView-style) → M-VIZ.advanced
- Viewport screenshot / PNG export → M-VIZ.advanced
- Server-side render fallback (for browsers without WebGL) → not planned
- User-customizable color map / scalar bar → bundled with M-VIZ.results
- Performance optimization (LOD, decimation) → triggered when fixtures > 1M faces

## Self-pass-rate estimate

**70%** — calibrated against:
- M5.0 frontend first-pass (~75% — small surface, ImportPage was new but conventional React)
- This milestone has slightly more risk because (a) vtk.js is a new dependency with non-trivial integration, (b) WebGL lifecycle bugs are a known frontend-pain class, (c) bundle size is a hard concern.

## Top failure modes anticipated

1. **vtk.js bundle bloats past 3 MB** → tradeoff: import only the modules we use, configure tree-shaking
2. **WebGL context leak across remounts** → required cleanup on unmount, regression test
3. **STL loader doesn't handle our fixtures** (e.g., ASCII vs binary, axis convention) → add fallback / clear error message
4. **Browser compat issue** (Safari WebGL quirks) → smoke on Chrome + Safari; document Edge/Firefox as untested-but-expected-OK
5. **vtk.js TypeScript types incomplete** → wrap with our own types; accept `any` boundaries documented inline
6. **Performance on large STLs** (naca0012 has 628 faces — fine; future imports could hit 100k+) → MVP doesn't optimize; deferred to M-VIZ.advanced
