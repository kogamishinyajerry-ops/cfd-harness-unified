## M5.0 Routine PR

### Goal

Ship workbench-side STL import that takes a user-uploaded STL, runs trimesh
sanity checks, scaffolds a minimal OpenFOAM case directory by forking the
LDC template, and routes the user into the existing `/workbench/case/{id}/edit`
flow.

### Non-goals (explicit)

STEP/IGES support · geometry healing · UI patch picking · cfMesh · NL · LLM
agent on case generation · TrustGate verdict policy · audit_package
filtering (the last two are M5.1).

### Path corrections (finding 7 applied)

Two v1 spec citations were wrong; corrected here for the surface scan to
cite the right files going forward:

- v1 said "Read `ui/backend/services/case_editor.py` end-to-end. Reuse its
  case-id allocator and draft-write path." → **Correct**: the
  case-editor route is at `ui/backend/routes/case_editor.py` (93 LOC); the
  service-layer counterpart with the case-id + draft persistence helpers is
  `ui/backend/services/case_drafts.py`. Reuse `case_drafts.py`'s allocator
  and the `ui/backend/user_drafts/` write-domain convention.
- v1 said template reference is `knowledge/case_profiles/lid_driven_cavity.yaml`
  → **Correct**: that directory does not exist. LDC profile lives at
  `knowledge/workbench_basics/lid_driven_cavity.yaml` (driver definition);
  gold-standard reference is `knowledge/gold_standards/lid_driven_cavity.yaml`.
  M5.0 forks the workbench_basics LDC profile + the corresponding template
  case directory (path TBD during code work; surface-scan it before the first
  `template_clone.py` test).

### Deliverables (≤ ~600 LOC main + ≤ ~250 LOC tests · slimmer than v1's 750+300 because TrustGate + audit_package work moves to M5.1)

#### A. Backend module · `ui/backend/services/geometry_ingest/`

- `stl_loader.py` — trimesh-based loader, returns canonical STL bytes + parsed solids
- `health_check.py` — watertight check, bbox, unit-guess (m/mm/in via bbox heuristic), solid count, face count, single-shell check
- `patch_detector.py` — extract per-solid name → patch list; honest "all defaultFaces" degradation message
- `__init__.py` — public API `IngestSTL(bytes) -> IngestReport`

#### B. Backend module · `ui/backend/services/case_scaffold/`

- `template_clone.py` — copy LDC template tree to `ui/backend/user_drafts/imported/{case_id}/`. Reuse the `case_drafts.py` allocator for `{case_id}`.
- `bc_injector.py` — write `triSurface/geometry.stl` + minimal `snappyHexMeshDict` stub referring to detected patches. Do NOT run sHM yet (that is M7's job).
- `manifest_writer.py` — emit `case_manifest.yaml` containing:
  - `source: "imported"`
  - `source_origin: "imported_user"` *(D3-routine portion: schema additive)*
  - `origin_filename`
  - `ingest_report_summary`
  - `created_at` (UTC ISO-8601)
  - `solver_version_compat`

  *Note*: the `source_origin: "imported_user"` field is M5.0 routine
  (additive schema change). The downstream **consumers** of this field
  — TrustGate hard-cap and audit_package filter — are M5.1 trust-core.
  M5.0 produces the field; M5.1 enforces policy on it.

#### C. Backend route · `ui/backend/routes/import_geometry.py`

- `POST /api/import/stl` — multipart upload, max 50 MB enforced in route handler
- 200 response → `{ case_id, ingest_report, edit_url }`
- 4xx response → honest fail message naming which sanity check broke

#### D. Frontend page · `ui/frontend/src/pages/workbench/ImportPage.tsx`

- Drop-zone or file picker (single STL)
- After upload: ingest-report card showing watertight ✓/✗, bbox, unit guess, patches detected, total faces
- WARN state (e.g. all-defaultFaces): inline help "export STL with named solids per inlet/outlet/wall"
- PASS or user-confirmed WARN: "Create case" button → `POST /api/import/stl` → redirect to `/workbench/case/{case_id}/edit`

#### E. Tests

- `ui/backend/tests/services/test_geometry_ingest.py`:
  - happy-path (named-solid sphere STL)
  - watertight-fail
  - named-solid-missing degradation
- `ui/backend/tests/services/test_case_scaffold.py`:
  - scaffold from LDC template asserts case_manifest schema (source, source_origin, origin_filename, ingest_report_summary, created_at, solver_version_compat fields all present and well-typed)
- `ui/backend/tests/routes/test_import_route.py`:
  - over-50MB rejection
  - end-to-end happy-path via TestClient

#### F. Dependency

Add `trimesh>=4.0` to `pyproject.toml` `[project.optional-dependencies].workbench`
(NOT default install — workbench extras only).

## M5.1 Trust-core micro-PR (deferred until after M5.0 has produced ≥1 real imported-case artifact)

### Goal

Enforce trust-core verdict policy on imported cases.

### Deliverables (~50 LOC main + ~30 LOC tests)

- **TrustGate hard-cap** in `src/metrics/trust_gate/`: imported cases (those with `case_manifest.source_origin == "imported_user"`) cap verdict at `PASS_WITH_DISCLAIMER`. No code path can return `PASS` for an imported case (no gold-standard reference exists).
- **audit_package filter** in `src/audit_package/`: default share output excludes imported cases; new CLI flag `--include-imported` opts in. Audit log still records the imported run for forensics.
- **Schema propagation**: `RunHistory` schema gains `source_origin` field (read from manifest at run-finalize time).
- **Tests**: imported case never returns `PASS`; `--include-imported` flag inclusion behavior verified.

### Governance gates (M5.1 trust-core path · per ROADMAP §"Governance posture: downgraded" trust-core list + DEC-V61-087 §4)

1. **DEC authored** — `DEC-V61-08X · M5.1 trust-core hard-cap for imported cases` (X = next available number; check `.planning/decisions/` at authoring time)
2. **Codex 3-round review** — full arc per RETRO-V61-001 trust-core trigger; CHANGES_REQUIRED on round 3 → escalate to Kogami
3. **Kogami high-risk-PR clearance** — strategic package authored:
   - `intent_summary.md` per DEC-V61-087 §4.4 schema (validator: `python3 scripts/governance/validate_strategic_package.py --intent <p> --risk <p>`)
   - `merge_risk_summary.md` per DEC-V61-087 §4.4 schema
4. **User explicit ratification** — final authority before merge
5. **Notion sync** — DEC + RETRO sync per `notion-sync-cfd-harness` skill workflow
