# M6 · gmsh-based unstructured meshing path — Kickoff Spec v2

> **Scope**: source-of-truth for what M6 actually delivers. Original brief
> (`brief_2026-04-27.md`) presented three open strategic questions; Kogami
> ruled (APPROVE_WITH_COMMENTS); this spec encodes the resolved scope.

## M6/M7 framing decision (the central call · Kogami finding 1)

Brief presented three reads (i parallel alternatives / ii sequential with
`mesh_engine` selector / iii M6=gmsh + M7=fill-in sHM stub). Kogami's
strategic assessment endorsed **read (iii)** as most coherent because:

1. M5.0 already shipped `system/snappyHexMeshDict.stub` expecting M7 to
   fill it in — read (iii) preserves that artifact's purpose.
2. ROADMAP §M7 phrasing "snappyHexMesh execution + mesh budget tiering"
   matches the "fill the stub" reading exactly.
3. A `mesh_engine` schema field with only one valid value at M6-close is
   premature; deferring it to M7 (when the second engine exists) avoids
   carrying a per-case selector in M6 that has nothing to select between.

**M6 scope going forward**: gmsh path produces a runnable mesh under
`user_drafts/imported/{case_id}/constant/polyMesh/` for every imported
case. The M5.0 sHM stub stays untouched in M6 (still oriented toward
M7's eventual sHM-engine path).

**M7 scope (forward-binding · binds future M7 brief)**: fill in the
sHM stub from M5.0 + execute snappyHexMesh + apply two-tier mesh budget
caps (D6) + introduce `mesh_engine` field if a per-case selector is
warranted at that point (defer the field's existence to M7 itself).

## M6.0 Routine PR

### Goal

Take an imported case (M5.0 output) and produce a runnable OpenFOAM
mesh under `user_drafts/imported/{case_id}/constant/polyMesh/` via
gmsh + `gmshToFoam`. After M6.0 lands, the imported case has a real
mesh that M7's `mesh_already_provided` executor patch can consume.

### Non-goals (explicit)

snappyHexMesh path · mesh-engine selector schema · trust-core executor
patches (M6.1) · per-case mesh quality auto-tuning · adaptive remeshing.

### Deliverables (~600 LOC main + ~250 LOC tests · gmsh wraps a lot of complexity in a small surface)

#### A. Backend module · `ui/backend/services/meshing_gmsh/`

- `gmsh_runner.py` — pure-Python `gmsh` API: open `triSurface/{stl}`,
  set characteristic length, generate 3D unstructured tetrahedral mesh,
  write `.msh` to a known path. Honors a `cell_count_target` hint (D6
  budget cap is enforced separately — see C below).
- `to_foam.py` — invoke `gmshToFoam` (OpenFOAM utility, runs in the
  existing CFD container) to convert `case_dir/imported.msh` →
  `case_dir/constant/polyMesh/{points,faces,owner,neighbour,boundary}`.
- `cell_budget.py` — D6 enforcement layer (see §"D6 cap calibration" below).
- `__init__.py` — public `mesh_imported_case(case_id, *, mesh_mode) -> MeshResult`.

#### B. Backend route · `ui/backend/routes/mesh_imported.py`

- `POST /api/import/{case_id}/mesh` body `{ mesh_mode: "beginner"|"power" }`
- 200 → `{ mesh_summary: { cell_count, face_count, point_count, mesh_mode_used, polyMesh_path } }`
- 4xx → honest fail (`failing_check ∈ {gmsh_diverged, cell_cap_exceeded, gmshToFoam_failed, case_not_found, source_not_imported}`)

#### C. D6 cap calibration (Kogami finding 2 · deferred-to-empirics)

D6 inherits 5M-beginner / 50M-power caps from M5 spec v2. Kogami noted
that 5M may be too tight for moderately-detailed STL geometry under
default gmsh settings. Resolution adopted from finding 2 recommendation:

- **M6.0 ships only the 50M power-mode hard cap** as a verdict-gate-grade
  guard (the 50M ceiling is a UX safety against runaway resource use).
- **Beginner-mode cap is initially soft** — a warning at 5M cells, the
  mesh still generates. M6.0 also emits per-fixture telemetry
  (`{cell_count, generation_time_s}` for each of the 3 bundled STLs and
  the user's first imported case) into `.planning/dogfood/M6.0_calibration.md`.
- **D6 calibration follow-up** (M6.0.1, ~20 LOC): the beginner soft-warning
  may be converted to a **hard cap** only when telemetry covers **N≥3
  distinct STLs** — a mix of the 3 bundled fixtures (`ldc_box`, `cylinder`,
  `naca0012`) AND real user uploads. Single-PR-run telemetry is anecdote
  (per Notion-Opus advisory 2026-04-27 finding 4c). Until N≥3 reached:
  - 5M beginner cap stays as **soft warning only**
  - 50M power-mode hard cap is **unaffected** (verdict-grade guard, not a
    calibrated value)
- Once N≥3 telemetry exists, the beginner hard cap is set at
  `ceil(2 × empirical_max(observed))` rounded to a clean integer.

This avoids the misfire failure mode where the user uploads a normal STL
and hits an unjustified 5M-cell rejection, AND avoids the symmetrical
mode where M6.0.1 ships a number with the same epistemic weight as the
5M it replaces.

#### D. Frontend page · `ui/frontend/src/pages/workbench/MeshWizardPage.tsx`

- Reachable from `/workbench/case/:caseId/mesh` (new route).
- Mesh-mode toggle (beginner default · power opt-in) — single-toggle UX
  per D6 phrasing "single toggle inside Mesh Wizard, not a separate page".
- "Generate mesh" button → POST + show progress (gmsh can take 30-300s
  on real geometry).
- On success: cell-count summary card + "Continue to run (M7)" link
  (link is disabled until M7 lands — copy: "M7 will land OpenFOAM run").
- On WARN (5M soft cap exceeded in beginner mode): inline note
  "Your mesh is larger than typical beginner sizing; consider switching
  to power mode" — non-blocking.

#### E. Tests · `ui/backend/tests/`

- `test_meshing_gmsh.py` (~6 cases): happy-path (cylinder.stl), gmsh
  divergence, 50M cap exceeded, gmshToFoam failure mocked, mesh result
  schema, per-fixture cell counts (golden values for the 3 M5 fixtures).
- `test_mesh_imported_route.py` (~5 cases): happy-path, source not
  imported (e.g., a whitelist case_id), case not found, mesh-mode
  validation, cap-exceeded path.

#### F. Dependencies · `pyproject.toml [project.optional-dependencies].workbench`

- Add `gmsh>=4.11,<4.13` (per D4)
- macOS arm64 wheel availability: verify via
  `uv pip install gmsh` smoke test on M-series before M6.0 lands;
  document conda-forge fallback path in `docs/install.md` if the PyPI
  wheel is missing.
- CI smoke install: add ubuntu-latest + macos-13 arm64 jobs that just
  run `uv pip install -e '.[workbench]'` + `python -c "import gmsh"`
  to catch wheel regressions early.

## M6.1 Trust-core micro-PR

### Goal

Allow `FoamAgentExecutor.execute()` to skip `blockMesh` when the case
already has a mesh under `constant/polyMesh/`.

### Deliverables (~25 LOC main + ~20 LOC tests · narrow scope per finding 4)

- Add `mesh_already_provided: bool` field to `task_spec` (defaults `False`).
- In `FoamAgentExecutor.execute()` at `src/foam_agent_adapter.py:697`,
  guard the `blockMesh` invocation:
  ```python
  if not task_spec.mesh_already_provided:
      blockmesh_log = self._run_in_container("blockMesh", ...)
  ```
- A second guard checks `(case_dir / "constant" / "polyMesh").exists()`
  defensively — if the flag was set but no mesh is present, fail fast
  rather than running with an empty mesh.
- M6.0's `to_foam.py` sets `mesh_already_provided=True` when invoking
  the wizard run path for an imported case.
- Tests: blockMesh-skipped on imported cases · blockMesh-still-runs on
  whitelist cases · honest failure when flag set but polyMesh missing.

### What's deliberately NOT in M6.1 (per finding 4)

- No `case_kind = "imported_user"` value in executor dispatch.
- No reading of `case_manifest.yaml` from inside the executor.
- No new line-A → trust-core data dependencies.

The `mesh_already_provided` flag is a structural boolean; the
`(case_dir / "constant" / "polyMesh").exists()` check is a filesystem
predicate. Both are trust-core-internal contracts — line-A-authored
content does not flow into executor decisions.

### Governance gates (M6.1 trust-core)

2. Codex 3-round review (multi-file? — actually single-file for M6.1).
3. Kogami high-risk-PR clearance with intent_summary + merge_risk_summary.
