# M6 Kickoff · Strategic Clearance Brief

## What this brief asks

M5.0 shipped (`23bcba6`). The next phase per ROADMAP §"Beginner Full Stack
M5–M8" is **M6 — gmsh-based unstructured meshing path**. Pre-implementation
surface scan (per DEC-V61-088 routine gate) found **clean name collisions**
but **two substantive strategic questions** that Kogami should resolve
before code work starts.

**Strategic-layer questions for Kogami**:

1. **Tier classification & line-A/line-B boundary**: M6's deliverable is a
   meshing pipeline that takes M5's `triSurface/{stl}` and produces an
   OpenFOAM-runnable `constant/polyMesh/` (or equivalent). The current
   `FoamAgentExecutor.execute()` at `src/foam_agent_adapter.py:697`
   **hardcodes a `blockMesh` invocation** with no skip-or-substitute
   hook. M6 cannot land as pure routine path without touching this
   line-B / trust-core surface. Possible splits:
   - (a) **M6.0 routine** (gmsh service produces `.msh` + writes
     `constant/polyMesh/` directly via `gmshToFoam`, all under
     `user_drafts/imported/{case_id}/`) + **M6.1 trust-core** (patch
     `FoamAgentExecutor.execute()` to skip `blockMesh` if
     `constant/polyMesh/` already populated) — analogous to M5.0/M5.1.
   - (b) **Single trust-core M6 PR** (DEC + 3-round Codex + Kogami) —
     anything touching the executor lives on trust-core.
   - (c) **M6.0 produces only a parallel `.msh` artifact**, no executor
     integration; M7 owns the executor patch as part of "real OpenFOAM
     run on imported case" — folds the line-B work into M7.

2. **M6 vs M7 relationship**: ROADMAP frames M6 as "gmsh meshing path
   (alternative to snappyHexMesh)" and M7 as "snappyHexMesh execution
   + mesh budget tiering". Are these:
   - (i) **Parallel alternatives** the user picks between (gmsh for
     unstructured/complex internal flows, sHM for external aero) — M6
     and M7 ship two different `mesher` engines, both gated by D6's
     `mesh_mode` budget caps?
   - (ii) **Sequential layering** — M6 lands gmsh first; M7 adds sHM
     as the second engine; mesh-engine selection is per-case per a
     new `mesh_engine` field?
   - (iii) **M6 = unstructured (gmsh) for arbitrary STL** + M7 = the
     snappyHexMesh stub from M5 finally being filled in (so M5's sHM
     stub is the M7 starting point, not an M6 input)?

This is **NOT a high-risk PR pre-merge clearance**. It's a phase-kickoff
strategic gate analogous to M5's, recorded under
`.planning/reviews/kogami/m6_kickoff_governance_clearance_2026-04-27/`.

## M6 phase scope (proposed for clearance)

### Origin and binding decisions inherited from M5

- **D4 (deferred from M5 kickoff per Kogami M5 finding 4)**: pin
  `gmsh>=4.11,<4.13`, CI smoke install matrix on ubuntu-latest +
  macos-13 arm64, macOS arm64 wheel known-issue posture with
  conda-forge fallback documentation.
- **D6 (binds M6 + M7 per spec v2)**: `mesh_mode="beginner"` (default)
  hard-cap **5M cells**; `mesh_mode="power"` cap **50M cells**. Single
  toggle inside Mesh Wizard, not a separate page. M6 must enforce
  these caps — the cell-count cap is a trust-grade UX guarantee
  (otherwise user fires off a 200M-cell run and the box dies).

### M5 → M6 handoff contract (already established)

`ScaffoldResult` from M5.0 produces, for each imported case:

```
user_drafts/imported/{case_id}/
    triSurface/{origin_filename}       ← gmsh's input
    system/snappyHexMeshDict.stub      ← deliberately stub'd; M6 likely ignores
    case_manifest.yaml                 ← contains source_origin=imported_user,
                                         ingest_report (bbox, patches, faces)
user_drafts/{case_id}.yaml             ← editor-facing case YAML
                                         (workbench-basics shape · solver=simpleFoam)
```

M6's gmsh meshing service consumes `triSurface_path` + reads
`case_manifest.yaml` for patch list, then writes the resulting mesh
under the same `imported_case_dir`.

### Proposed deliverables (subject to Kogami's tier-classification verdict above)

#### Path-(a) split: M6.0 routine + M6.1 trust-core

**M6.0 routine (~700 LOC main + ~250 LOC tests)**:

- **A. Backend module** · `ui/backend/services/meshing_gmsh/`:
  - `gmsh_runner.py` · pure-Python gmsh API to read STL → unstructured
    tetrahedral mesh; respects mesh_mode cell-count cap (D6)
  - `to_foam.py` · invoke `gmshToFoam` (OpenFOAM utility, runs in the
    existing CFD container) to convert `.msh` → `constant/polyMesh/`
    inside the imported case dir
  - `cell_budget.py` · two-tier budget enforcer (5M / 50M caps per D6)
  - `__init__.py` · public `mesh_imported_case(case_id, mesh_mode) -> MeshResult`
- **B. Backend route** · `ui/backend/routes/mesh_imported.py`:
  - `POST /api/import/{case_id}/mesh` — body `{ mesh_mode: "beginner"|"power" }`
  - 200 → `{ mesh_summary: {cell_count, face_count, volume, ...} }`
  - 4xx → honest fail (cell-count exceeded, gmsh divergence, etc.)
- **C. Frontend page** · `ui/frontend/src/pages/workbench/MeshWizardPage.tsx`:
  - Mesh-mode toggle (beginner/power per D6)
  - "Generate mesh" button → POST + show progress + cell-count summary
  - On success: enable "Continue to run" (M7 territory)
- **D. Tests** · `ui/backend/tests/test_meshing_gmsh.py`,
  `test_mesh_imported_route.py`
- **E. Dependencies** · `pyproject.toml [project.optional-dependencies].workbench`:
  - `gmsh>=4.11,<4.13` (per D4)
  - `meshio>=5.3` (read/write `.msh`/foam mesh formats)
  - macOS arm64 wheel known-issue documented; conda-forge fallback in install docs
- **F. Fixtures** (extend `examples/imports/`):
  - Pre-meshed reference for `cylinder.stl` and `naca0012.stl` (small,
    deterministic) for golden-mesh regression checks

**M6.1 trust-core (~50 LOC main + ~30 LOC tests)**:

- Patch `FoamAgentExecutor.execute()` to skip `blockMesh` when
  `case_dir/constant/polyMesh/` is already populated (gmshToFoam wrote
  it). Add `mesh_already_provided: bool` task_spec field; the imported-
  case dispatch path sets it.
- Update `_generate_lid_driven_cavity` and the case-kind dispatch to
  recognize `case_kind = "imported_user"` (a new case_kind value that
  reads from `case_manifest.yaml` at runtime).
- Tests asserting blockMesh-skip on imported cases.

**M6 governance gates**:

- M6.0 routine path · single-pass Codex review · direct commit to main · no DEC
- M6.1 trust-core · DEC-V61-08X · Codex 3-round · Kogami high-risk-PR clearance

## Pre-implementation surface scan findings

### ✅ Clean

- `grep -RIn "gmsh|meshio|gmshToFoam"` returns only one match in
  `scripts/gen_skill_index.py:63` (a heuristic skill-categorizer string,
  not a live reference). No naming collisions.
- `meshing_gmsh/`, `mesh_imported`, `MeshWizardPage` are unused names.
- M5.0's `ScaffoldResult` already exposes `triSurface_path` + `imported_case_dir`
  → handoff contract clean.
- `case_manifest.yaml` already exposes `source_origin: imported_user` +
  patch list → M6 can consume without schema migration.

### 🔴 Substantive contradiction (needs strategic resolution)

**Contradiction · `FoamAgentExecutor` is line-B / trust-core**:

ROADMAP §"Governance posture: downgraded" explicitly lists
`src/foam_agent_adapter.py` under trust-core. The executor at
`src/foam_agent_adapter.py:697` hardcodes `blockMesh`. Any M6 path that
wants gmsh-generated mesh to actually drive a real solver run MUST
modify this surface. Three reads in the §"Strategic-layer questions"
section above; Kogami's call.

### 🟡 Open-but-not-blocking

**macOS arm64 gmsh wheel availability**: D4 anticipated this. The
official `gmsh` PyPI wheel does ship arm64 macOS as of 4.11+, but
historically the wheel was missing. Verification step recommended:
test `uv pip install gmsh` on M-series before M6 lands; document the
conda-forge fallback path in `docs/install.md`.

**meshio dependency**: gmsh's Python API CAN write OpenFOAM polyMesh
directly via `gmsh.write("case.foam")` — meshio may be redundant. Open
question for the implementer; not strategic.

**Existing M5.0 `snappyHexMeshDict.stub`**: M5.0 produced this stub
expecting M7 to fill it in. If M6 lands gmsh-as-default-mesher, the
stub becomes orphaned for imported cases that take the gmsh path.
Strategic question: is the stub even relevant given M6's existence?
Likely yes (if M7's sHM path is opt-in), but should be addressed.

## What this brief is asking Kogami

Per Kogami's role (strategic, not code-level):

1. **Tier classification**: which of the three reads (a) / (b) / (c) for
   the line-A / line-B boundary is strategically coherent with the v6.2
   three-layer architecture? Specifically:
   - Is path-(a) (M6.0 routine + M6.1 trust-core) the right precedent
     given M5.0/M5.1 already used this split?
   - Or does path-(c) (defer all executor work to M7) better honor the
     "M7 = real OpenFOAM run" framing in ROADMAP §M7?
2. **M6 vs M7 relationship**: of (i) / (ii) / (iii), which framing is
   coherent? ROADMAP says "M6 = gmsh ... alternative to snappyHexMesh"
   and "M7 = snappyHexMesh ... mesh budget". Need a ruling on whether
   `mesh_engine` becomes a per-case selector or whether the gmsh and
   sHM paths are sequenced in time without user choice.
3. **D6 cell-count cap enforcement**: should the 5M / 50M caps be
   enforced **inside the gmsh service** (M6.0 routine), or via a
   trust-core gate (M6.1), given that exceeding the cap is a UX guard
   not a verdict guard?
4. **Locked-decision coherence**: D4 (gmsh dep pin + CI matrix +
   conda-forge fallback) — is pinning a Python lib + adding a CI matrix
   a routine-path action, or does pinning (which is reversible but
   cross-cutting) warrant heavier review?
