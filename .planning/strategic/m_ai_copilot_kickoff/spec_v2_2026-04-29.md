## §B · API contracts

### B.1 `face_annotations.yaml` schema (canonical)

```yaml
schema_version: 1
case_id: <string>
revision: <int>             # bumped on every PUT; AI envelope echoes back
last_modified: <ISO8601>
faces:
  - face_id: <string · "fid_" + 16-hex>
    name: <string · user-given or AI-suggested>
    patch_type: "patch" | "wall" | "symmetry" | "empty" | "cyclic"
    bc:                         # field → BC dict; mirrors OpenFOAM 0/<field> shape
      U: { type: "fixedValue", value: "(...)" }
      p: { type: "zeroGradient" }
      ...
    physics_notes: <string · free-form>
    confidence: "user_authoritative" | "ai_confident" | "ai_uncertain"
    annotated_by: "human" | "ai:<source>"
    annotated_at: <ISO8601>
```

Invariants:
- `revision` is monotonic. The PUT route rejects a write whose `if_match_revision` doesn't match the current.
- `face_id` is a sha1 of the face's vertex-coordinate-sorted-tuple, hex-truncated to 16 chars + `fid_` prefix. Stable under deterministic mesh regen.
- `confidence == "user_authoritative"` is sticky: AI on subsequent runs MUST NOT overwrite without the user re-running through the dialog panel.

### B.2 `AIActionEnvelope` schema

```python
class UnresolvedQuestion(BaseModel):
    id: str                    # stable across retries (deterministic from face_ids + question type)
    kind: Literal["face_label", "physics_value", "boundary_type", "free_text"]
    prompt: str                # human-readable
    needs_face_selection: bool
    candidate_face_ids: list[str] = []   # if AI has narrowed the candidate set
    candidate_options: list[str] = []    # for boundary_type / patch_type picks
    default_answer: str | None = None    # AI's best guess if forced

class AIActionEnvelope(BaseModel):
    confidence: Literal["confident", "uncertain", "blocked"]
    summary: str               # one-sentence what-happened
    annotations_revision_consumed: int   # which annotations file revision the AI ran against
    annotations_revision_after: int      # the revision after AI's writes (no-op if AI didn't write)
    unresolved_questions: list[UnresolvedQuestion] = []
    next_step_suggestion: str | None = None
    error_detail: str | None = None      # only when confidence == "blocked"
```

The route returns this envelope at HTTP 200 when the AI action ran successfully (regardless of confidence). True 4xx still happens on input validation; transport errors still happen on infrastructure failure.

### B.3 `/api/cases/<id>/face-annotations` route

| Method | Body | Response | Notes |
|---|---|---|---|
| GET | — | `face_annotations.yaml` parsed | 404 if no annotations file yet (UI treats as empty) |
| PUT | `{ if_match_revision, faces: [...] }` | new state + new `revision` | 409 on revision mismatch (concurrent edit) · 422 on unknown face_id · 200 on success |

### B.4 `setup-bc` route envelope mode

Existing `POST /api/import/<case_id>/setup-bc` keeps its current behavior for the LDC fixture (return `SetupBcSummary`). New query parameter `?envelope=1` opts into the envelope shape:

- 200 + `AIActionEnvelope { confidence: "confident", ... }` on success.
- 200 + `AIActionEnvelope { confidence: "uncertain", unresolved_questions: [...] }` when partial.
- 200 + `AIActionEnvelope { confidence: "blocked", ... }` when AI cannot proceed.

Frontend opts in. Backward-compatible.

### B.5 face_id hash function

```python
def face_id(face_vertices: list[tuple[float, float, float]]) -> str:
    """Stable hash for an OpenFOAM polyMesh face. Independent of vertex
    ordering within the face — sorts coordinates first.
    """
    rounded = [tuple(round(c, 9) for c in v) for v in face_vertices]
    sorted_verts = sorted(rounded)
    h = hashlib.sha1(repr(sorted_verts).encode("utf-8")).hexdigest()
    return f"fid_{h[:16]}"
```

Invariance: deterministic gmshToFoam → identical face_id across regens. The 9-decimal rounding absorbs floating-point drift.

## §C · Frontend interaction state machine

```
StepStatus: pending → active → [completed | error | awaiting_user]
                                                     ↓
                                          DialogPanel renders questions
                                                     ↓
                                              user answers all
                                                     ↓
                                          [继续 AI 处理] enabled
                                                     ↓
                                              re-run AI action with annotations
                                                     ↓
                                            confident → completed
                                            uncertain → awaiting_user (with new questions)
                                            blocked   → error
```

`awaiting_user` is a NEW StepStatus. Visual: amber clock icon in StepTree; status strip shows "Step N · awaiting your input".

## §D · Concrete annotation panel flow (Tier-A demo)

The Tier-A demo on LDC uses a **forced** `?envelope=1&force_uncertain=1` query so we can dogfood the flow without needing real AI ambiguity:

1. User clicks `[AI 处理]` on Step 3.
2. Backend setup-bc returns:
   ```json
   {
     "confidence": "uncertain",
     "summary": "Set up LDC defaults; please confirm lid orientation.",
     "annotations_revision_consumed": 0,
     "annotations_revision_after": 1,
     "unresolved_questions": [
       {
         "id": "lid_orientation",
         "kind": "face_label",
         "prompt": "Confirm which face is the moving lid (top, +z).",
         "needs_face_selection": true,
         "candidate_face_ids": ["fid_abc...", "fid_def..."],
         "default_answer": "fid_abc..."
       }
     ]
   }
   ```
3. DialogPanel renders the question. User clicks the lid face in the viewport. AnnotationPanel pops up at click position with `name`, `patch_type` pre-filled. User saves.
4. PUT `/face-annotations` writes to disk, returns `revision: 2`.
5. User clicks `[继续 AI 处理]`. Re-runs setup-bc with `if_match_revision=2`. Returns `confident`.

This demo path also works as the integration test.

## §F · Failure modes (what Codex will key on)

| Class | Mode | Mitigation |
|---|---|---|
| Concurrency | User edits annotation while AI is mid-run | `revision` conflict check on both PUT and AI envelope |
| Stability | mesh regen changes face_id | rounded coordinates; regen test in Step 2 |
| Race | DialogPanel renders stale questions after a fresh run | `genRef`-style guard on the StepPanel side, mirrors V61-097 SolveStreamContext |
| Persistence | AI overwrites user_authoritative annotation | strict invariant in the merge function; test asserts user-authoritative wins |
| UX | User clicks a face but the annotation panel covers the geometry | panel positioned at click coords with smart anchoring (avoid covering the picked face) |
| 4xx surface | unknown face_id in PUT body | 422 with detailed error · test |
| Backward-compat | legacy setup-bc consumers without `?envelope=1` | unchanged response shape · test |
| Hidden state | annotation file isn't part of audit package | extend audit package to include it (separate Tier-A line item) |
