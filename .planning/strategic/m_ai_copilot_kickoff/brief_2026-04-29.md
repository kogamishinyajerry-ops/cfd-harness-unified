## TL;DR

The Workbench can now drive a **fixed** geometry (the LDC cube) end-to-end. To drive an **arbitrary** STL (the Pivot Charter Addendum 2 promise) the AI side cannot stand alone — it lacks geometric semantic information that a human sees instantly. Every prior milestone deferred this gap. M-AI-COPILOT closes it by treating the **human-AI collaboration layer** as the core deliverable: AI signals uncertainty, human annotates by clicking on the 3D model, and the annotation persists as case metadata for downstream steps to consume.

This consumes the §11.1 BREAK_FREEZE quota's last slot (2/3 → 3/3 per V61-096 §Governance projection).

## Why now (the architectural insight)

CFDJerry's call 2026-04-29:

> 任意STL的AI处理非常困难，必须有一套优秀的人机交互机制，确保AI在完全不知道怎么办的情况下，可以及时获取人类工程师的指导，人类工程师也可以自己对渲染出来的CAD模型进行点击、编辑、命名……

This refutes two simpler framings that earlier milestones implicitly assumed:

1. **"AI alone can drive the workflow"** — M-PANELS Tier-A's `[AI 处理]` button suggests an autonomous step. The LDC fixture works because lid/walls/front-back patches are knowable by axis-aligned face geometry. For an arbitrary STL with N inlet/outlet/wall surfaces, no purely-geometric heuristic disambiguates them — the AI must ask.
2. **"Face picking is a polish item"** — V61-096 deferred face-pick to M-PANELS.advanced. But without face-pick, there is no channel for the human to answer the AI's questions. M-PANELS.advanced was the missing primitive, not a nice-to-have.

So M-AI-COPILOT and the deferred M-PANELS.advanced face-pick **must merge**. Doing them sequentially would mean shipping AI escalation hooks against a UI surface that doesn't exist yet, then doing the surface, then re-wiring.

## The four interaction primitives

These are the contract surface for collab-first AI. Each must be implementable in isolation but compose into the full workflow.

### 1. AI uncertainty signal (`confidence` enum + `unresolved_questions[]`)

Every AI action returns one of:
- **`confident`**: AI completed the step, applied changes, ready for `[下一步]`.
- **`uncertain`**: AI completed a partial / fallback action, but flagged ambiguities. User can ratify, override, or escalate to "blocked".
- **`blocked`**: AI cannot proceed without explicit human input. Step status flips to a new `awaiting_user` state.

When non-`confident`, the AI returns a structured list of unresolved questions:

```typescript
interface UnresolvedQuestion {
  id: string;                  // stable across retries
  kind: "face_label" | "physics_value" | "boundary_type" | "free_text";
  prompt: string;              // human-readable, e.g. "Which face is the inlet?"
  needs_face_selection: boolean;  // true → user must click on the 3D viewport
  candidate_options?: string[];   // optional pre-populated picks (e.g. patch type enum)
  default?: string;            // AI's best guess if forced to pick
}
```

The Step 4 task panel renders these questions inline; answering them dispatches a follow-up call back into the AI action.

### 2. 3D viewport face-pick (vtk.js cell-picker + face-id stable mapping)

The shared Viewport gains a `pickMode` prop. When set:
- Hover a face → highlight the patch (semi-transparent yellow overlay) + tooltip showing current annotation (if any).
- Click → selects that face; a floating panel appears at the click position with annotation fields.
- Shift-click → multi-select (for "all walls" assignments).

Backend exposes a stable `face_id`: a hash over the polyMesh face's vertex-set so it survives mesh regen as long as topology doesn't change.

### 3. Annotation persistence (`face_annotations.yaml`)

```yaml
# <case_dir>/face_annotations.yaml
schema_version: 1
case_id: <case_id>
created_by: human | ai | merged
faces:
  - face_id: "0x7f3a..."
    name: "inlet"            # user-given or AI-suggested
    patch_type: "patch"      # OpenFOAM polyBoundaryMesh type
    bc:
      U: { type: "fixedValue", value: "(1 0 0)" }
      p: { type: "zeroGradient" }
    physics_notes: "free-form user notes"
    confidence: "confident" | "uncertain" | "user_authoritative"
    annotated_at: 2026-04-29T...
    annotated_by: human | "ai:gpt-4" | "ai:rule-based"
```

Subsequent AI steps READ this file before deciding; user-authoritative entries override AI guesses. The file is part of the case dir → travels with import / export / audit packages.

### 4. Bidirectional dialog panel

Right-rail task panel split into two zones:
- **Top half**: Step's primary action (existing `[AI 处理]` button + status).
- **Bottom half**: AI's `unresolved_questions[]` rendered as a checklist. Each question has an inline answer affordance (text input / face-picker trigger / dropdown). Answering all questions arms a `[继续 AI 处理]` button that re-runs the step's AI action with the new annotations in scope.

Status strip surfaces a persistent "AI is waiting for: 3 face labels" hint when applicable.

## User stories (concrete)

### Story A · Engineer imports a flange STL with 5 surfaces
1. Step 1 (Import): mesh imports, viewport shows the flange in 3D.
2. Step 2 (Mesh): user clicks `[AI 处理]`. AI generates mesh, returns `confident`.
3. Step 3 (Setup BC): user clicks `[AI 处理]`. AI returns `blocked` with 5 unresolved questions:
   - "Which face is the inlet?" (needs_face_selection: true)
   - "Which face is the outlet?" (needs_face_selection: true)
   - "These 3 faces look like walls — is that correct?" (with face_id list)
4. User clicks faces in the viewport, names them, sets BCs. Annotations save to `face_annotations.yaml`.
5. User clicks `[继续 AI 处理]`. AI re-runs setup-bc reading the annotations, returns `confident`.
6. Step 4 (Solve), Step 5 (Results): proceed normally.

### Story B · Engineer overrides AI guess on a known geometry
1. Step 3 (Setup BC) on the LDC cube. AI returns `confident` (lid + walls).
2. User clicks the lid face anyway, opens annotation panel, changes `physics_notes` to "moving wall, U=(0.8, 0, 0)".
3. The annotation file records the override; AI on next call reads `user_authoritative` and respects it.

### Story C · AI flags a contradiction
1. User has manually labeled face X as "inlet". User then clicks `[AI 处理]` on Step 4 (Solve) before completing Step 3.
2. AI returns `uncertain` because the label is incomplete (no velocity assigned). Question: "Inlet face needs a U value — what's the inlet velocity?"

## Tier scope

### Tier-A · LDC + simple-cube fixture (single-PR scope)

Mandatory in this milestone:
- The 4 interaction primitives wired end-to-end.
- Backend: `face_annotations.yaml` reader/writer + face_id hash + AI-action wrapper that returns the structured `confidence + questions` envelope.
- Frontend: vtk.js cell-picker, annotation panel, dialog panel split, `awaiting_user` step state.
- One AI demonstration: Step 3 (Setup BC) on the LDC fixture **forced** into a `blocked` state via a flag, so we can dogfood the dialog flow without arbitrary STL.
- Tests: face_id stability under polyMesh regen · annotation round-trip · dialog state machine · Codex pre-merge.

### Tier-B · Arbitrary STL with rule-based AI (next milestone candidate)

Deferred:
- Real AI integration that genuinely says "I don't know" on imported geometry — needs a rule-based classifier (axis-alignment heuristics + face-area clustering) as a baseline, with explicit confidence gating. Step 2/3/4 each get a Tier-B AI implementation.
- LLM-mediated dialog (chat input box) — defer to P3+ per Pivot Charter §1.

### Tier-C · Polish

- Multi-select rectangle drag, undo/redo on annotations, annotation diff visualization, annotation-driven mesh region naming.

## Open questions (pre-impl scan will close)

1. **face_id stability**: vertex-set hash works as long as `gmshToFoam` is deterministic. Is it? (probable yes; verify with a regen test.)
2. **vtk.js picker latency**: at ~25k LDC cells the picker should be sub-frame; arbitrary STL with 1M cells may need vtkPropPicker → vtkCellPicker fallback or progressive picking. Tier-A focuses on LDC; Tier-B adds the perf knob.
3. **Race between user annotation and AI action**: if user is editing annotation while AI is mid-run, the AI's answer arrives stale. Need `annotation_revision` field similar to React's `genRef` — AI envelope includes the revision it ran against.
4. **Reset semantics**: re-running setup-bc with new annotations — full clean-slate or incremental? Tier-A: full re-run, simplest. Tier-B may want incremental.
