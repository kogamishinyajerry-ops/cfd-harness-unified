## Post-R3 defects (RETRO-V61-053 addendum class)

### Defect #1 — pressure_recovery Cp p_ref convention mismatch (DEFERRED)

**Class**: methodology gap
**Surfaced by**: Stage B v2 live OpenFOAM run
**Symptom**: `pressure_recovery.delta = 0.125` vs gold 1.00 (87.5% miss).
Mediating values:
- p_inlet (raw OpenFOAM kinematic pressure at x≈-9.77, area-averaged): -0.0623
- p_outlet (raw OpenFOAM kinematic pressure at x≈29.77): 0.0001
- Cp_inlet (extractor: (p_inlet - 0) / (0.5·1·1²)): -0.125
- Cp_outlet: 0.0001
- delta: 0.125

**Root cause**: Le/Moin/Kim 1997's reported Cp_inlet=-0.90 / Cp_outlet=
0.10 / delta=1.00 anchor uses a specific p_ref convention that V61-067's
extractor does not replicate. OpenFOAM kinematic-pressure incompressible
solvers anchor p ≈ 0 at the outlet (zeroGradient). Le/Moin/Kim's Cp_outlet
=+0.10 implies their p_ref is taken at some upstream-of-step station OR
the recovered downstream plateau itself, not at the outlet face. Without
the source paper's exact p_ref definition, the magnitudes don't align —
even though the SHAPE (positive recovery downstream) is correct.

**Why Codex static review wouldn't catch it**: this is a literature-
interpretation gap, not a code defect. Code is methodologically sound;
the gold ref_values come from a paper convention the harness doesn't
explicitly model.

**Disposition**: per V61-066 R3 precedent (separate methodology
correctness from physics fidelity), DEFER to DEC-V61-069. Two fix paths:
- Path A: derive p_ref from the run (e.g. mid-channel y/H=4.5 average)
  and apply BEFORE Cp computation; align with Le/Moin/Kim convention.
- Path B: pivot the gold ref_value to a derivable convention
  (e.g. delta-only, dropping inlet/outlet absolute Cp — they're
  mathematically meaningless without a fixed p_ref agreement).

**Methodology patch queue**: add `p_ref_convention_source` risk flag to
future intakes that gate on Cp scalars. Force the intake to declare:
"Is p_ref derivable from the run, or is it a literature convention?"

### Defect #2 — velocity_profile y-target nearest-cell offset (DEFERRED)

**Class**: methodology gap (mesh-resolution sensitivity)
**Surfaced by**: Stage B v2 live OpenFOAM run
**Symptom**: `velocity_profile_reattachment` 12.3% miss on point 0 and
-10.2% miss on point 2:

| target y/H | actual y/H | u_Ubulk extracted | gold | rel_error |
|---|---|---|---|---|
| 0.5  | 0.5125 | 0.449 | 0.40 | +12.3% |
| 1.0  | 0.9875 | 0.873 | 0.85 | +2.7% (PASS) |
| 2.0  | 1.75   | 0.943 | 1.05 | -10.2% |

**Root cause**: V61-052 7360-cell mesh has y-cells centred at non-
target positions. The B1/B2 interface at y=1 puts a cell centre at
0.9875 (just below) and 0.5125 (mid-B1). For y=2, the nearest cell
centre is 1.75 — a 0.25 mesh offset that shifts u_x noticeably in the
recovering shear layer.

`extract_velocity_profile_at_x` currently picks the SINGLE nearest cell
(min(|cy - y_target|)). On a fine mesh this is fine; on the V61-052
quick-run mesh, the resulting position errors translate to u_x errors
just outside the 10% tolerance.

**Why Codex static review wouldn't catch it**: extractor logic is
correct (and unit-tested at tests/test_bfs_extractors.py:test_picks_
nearest_y_when_no_exact_match). The gap is mesh-resolution-dependent.

**Disposition**: per V61-066 R3 precedent, DEFER to DEC-V61-070. Fix
paths:
- Path A: linear interpolation between bracketing cells. For y=2.0
  with cells at y=1.75 and y=2.x, interpolate u_x linearly. Trivial
  to add to extract_velocity_profile_at_x; reuses cells already in
  the column.
- Path B: 1500-iter mesh-refinement study (V61-052 mesh-independence
  follow-up).

**Methodology patch queue**: add `mesh_resolution_at_observable_x_y`
risk flag for any observable that samples interior cells at literature-
specified (x/H, y/H) coordinates.
