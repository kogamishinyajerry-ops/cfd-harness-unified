# DEC-V61-003: Land Path B Phase 1..4 (Case Editor + Decisions Queue + Run Monitor + Dashboard) as single rolling PR

## Decision summary

Under DEC-V61-002 Kogami elected Path B and delegated full development
decision authority through MVP completion. Phase 0 landed as PR #2
(merge SHA 6ae6d0b5). Kogami's follow-up — "continue until all
remaining planning tasks are done, enough for direct hands-on
experience" — is interpreted under Sole Primary Driver autonomous
authority as:

> Ship Phase 1..4 together on a single rolling branch `feat/ui-mvp-
> phase-1-to-4` as PR #3, regular merge commit, with end-to-end local
> experience guaranteed: `pip install -e .[ui,dev]` →
> `(cd ui/frontend && npm install)` → `./scripts/start-ui-dev.sh` →
> `http://127.0.0.1:5173`.

Phase 5 (Audit Package Builder) remains deferred per DEC-V61-002 §
Path-B interaction: it re-enters the critical path only after Q-1
(DHC gold-reference accuracy) and Q-2 (R-A-relabel pipe→duct)
external-Gate resolutions, because a commercial audit-package signer
cannot emit signed bundles while two known gold-accuracy /
whitelist-correctness issues are open.

### Phase 1 — Case Editor (Screen 2)
- **Backend**: `ui/backend/services/case_drafts.py` (+ `schemas/editor.py`,
  `routes/case_editor.py`) writes to `ui/backend/user_drafts/<case_id>.yaml`
  **only**. `knowledge/whitelist.yaml` and `knowledge/gold_standards/**`
  are mounted read-only (禁区 #1, #2 preserved).
- **Path-traversal guard** in `_draft_path()` — filename regex restricts
  to `[a-zA-Z0-9_-]+`.
- **Lint**: server-side `yaml.safe_load` + emission of warnings for
  missing `id`, `name`, `flow_type`, `geometry_type`,
  `turbulence_model`; client-side identical lint via `js-yaml` with
  180ms debounce.
- **Frontend**: `CaseEditorPage.tsx` uses CodeMirror 6 via
  `@uiw/react-codemirror` + `@codemirror/lang-yaml` (replaced Monaco
  for 4× smaller bundle footprint). Save/Revert buttons gated by
  `lint.ok && isDirty`; origin pill (draft / whitelist) and 禁区 guard
  card visible at all times.
- **Routes**: `GET /api/cases/{id}/yaml`, `PUT /api/cases/{id}/yaml`,
  `DELETE /api/cases/{id}/yaml` (revert), `POST /api/cases/{id}/yaml/lint`.

### Phase 2 — Decisions Queue (Screen 5)
- **Backend**: `ui/backend/services/decisions.py` parses
  `.planning/decisions/*.md` frontmatter + body; column classification
  is:
  - **Superseded** — frontmatter `superseded_by` set
  - **Closed** — filename suffix `_self_approve` / `_fuse` OR
    status field contains `closed`
  - **Accepted** — frontmatter `notion_sync_status` starts with `synced`
  - **Open** — everything else
  Q-3 is hardcoded CLOSED (resolved 2026-04-19 / re-closed 2026-04-20).
- **External-Gate queue**: parsed from `.planning/external_gate_queue.md`
  with `_Q_HEADER_RE` (allows `~~` strike-through for closed items).
- **Frontend**: `DecisionsQueuePage.tsx` is a 4-column Kanban
  (grid-cols-[1fr_1fr_1fr_1fr_320px]) with External-Gate sidebar
  showing Q-1 / Q-2 / Q-3 status.
- **Routes**: `GET /api/decisions`.

### Phase 3 — Run Monitor (Screen 3)
- **Backend**: `ui/backend/services/run_monitor.py` defines
  `RunProfile(target_iters=180, checkpoint_every=40,
  decay_log10=(("Ux",2.2),("Uy",2.0),("p",1.6)),
  initial_log10=-0.3, tick_ms_range=(70,160))`. Residual sampler:
  `10**(initial - decay*(iter/100) + Gaussian(0,0.18))` with plateau
  bumps every 27 iterations to simulate stall events. SSE stream
  emits `data: {json}\n\n` with phases `init` → `linear_solver` /
  `checkpoint` → `done`.
- **Frontend**: `RunMonitorPage.tsx` uses `EventSource` via
  `api.runStreamUrl(caseId)` + inline-SVG `ResidualChart.tsx`
  (LOG10_MIN=-12, LOG10_MAX=0.5; Ux #60a5fa / Uy #f472b6 / p #facc15).
  No external chart lib — preserves bundle size.
- **Routes**: `GET /api/runs/{id}/stream` (StreamingResponse
  `text/event-stream`, `X-Accel-Buffering: no`),
  `GET /api/runs/{id}/checkpoints`.

### Phase 4 — Dashboard (Screen 1)
- **Backend**: `ui/backend/services/dashboard.py` aggregates cases
  (per-case contract status via existing auto_verifier path),
  external-gate queue, decision timeline (sorted newest-first),
  summary counters (total/pass/hazard/fail/unknown cases;
  open/closed gates; accepted/closed/open/superseded decisions),
  current_phase (scraped from `STATE.md` `current_phase:` line),
  autonomous_governance_counter (v6.1-era count).
- **Frontend**: `DashboardPage.tsx` — 4-up StatCard row + 10-case
  physics-contract matrix with per-row Report/Edit/Run drill-down
  links + External-Gate list + Decision timeline with dot markers.
- **Routes**: `GET /api/dashboard`.

## Why

### Why single rolling branch instead of four PRs?

- **Shared shell**: Layout nav, API client, Tailwind primitives, and
  TypeScript type barrel are edited by all four phases. Four PRs would
  each need to touch them in a way that looks like merge-conflict bait.
- **User objective**: "足够让我直接上手体验" — Kogami asked for a
  hands-on experience, not a PR-review experience. Ship-ability of
  **the whole product surface** is the acceptance criterion, not
  ship-ability of individual phases.
- **Reversibility preserved**: The entire Phase 1..4 landing is still
  `git revert -m 1 <sha>` away from reversion; per-phase revert is not
  meaningfully more atomic given the shared nav/types surface.

### Why defer audit package?

Audit-package signing is the commercial critical-path feature — it is
what regulated-industry buyers procure. Shipping it before Q-1/Q-2 are
resolved would mean signing audit packages over known gold-accuracy /
whitelist-correctness issues, which directly contradicts the Path B
thesis. Phase 5 must wait for external Gate.

### Why CodeMirror 6 instead of Monaco?

Monaco's minified bundle is ~2.3MB (gzipped ~650KB). CodeMirror 6 +
`@codemirror/lang-yaml` is ~200KB gzipped. For a YAML-only editor, the
capability gap is negligible (both have syntax highlighting, bracket
matching, line numbers, theming). Monaco's IntelliSense advantage would
require a JSON Schema for case YAML — schema authoring is out of scope
for MVP. CodeMirror 6 also has cleaner React integration via
`@uiw/react-codemirror` (official wrapper).

### Why SSE instead of WebSocket for Run Monitor?

SSE is unidirectional (server → client) which matches residual-streaming
perfectly. FastAPI's `StreamingResponse` with `media_type="text/event-
stream"` is one wrapper call; WebSocket requires full connection
lifecycle management, auth negotiation, and reconnect logic. For Phase
3 MVP scope (streaming residuals + checkpoint metadata from a
MockExecutor), SSE is strictly simpler with no loss of function. If
Phase 5+ needs bidirectional (e.g., operator pause/resume), WebSocket
becomes warranted.

## Alternatives considered

- **Four separate PRs, one per phase.** Rejected per "why single rolling
  branch" above. Chief concern: nav shell + API client would churn
  across four PRs in ways that invite review fatigue and merge-conflict
  artifacts.
- **Include Phase 5 audit-package in this landing.** Rejected: Q-1/Q-2
  unresolved means any signed audit package would be emitted over
  known gold-accuracy / whitelist-correctness issues — directly
  contradicts Path B product thesis.
- **Monaco + monaco-yaml for Phase 1 (as listed in DEC-V61-002 phase
  table).** Rejected in favor of CodeMirror 6 per bundle-size analysis
  above. DEC-V61-002 table entry is superseded at the editor-library
  level (but Phase 1 scope is unchanged).
- **WebSocket for Phase 3 residual streaming (as listed in DEC-V61-002
  phase table).** Rejected in favor of SSE per "why SSE" above.
  DEC-V61-002 table entry is superseded at the streaming-transport
  level (but Phase 3 scope is unchanged).
- **VTK.js mesh viewport in Phase 3 (as listed in DEC-V61-002 phase
  table).** Deferred to Phase 5 audit-package work — would add ~1.5MB
  to bundle and is not needed for residual-stream demo. DEC-V61-002
  table entry for Phase 3 VTK is deferred, not cancelled.
