# DEC-V61-090 · M6.1 · `mesh_already_provided` flag + blockMesh skip

## Why

M6.0 (commit `e207258` · 2026-04-28 · Codex round-6 APPROVE recorded
at `reports/codex_tool_reports/m6_0_round6_postfix_*.log`) produces
`constant/polyMesh/` for imported user cases via gmsh + gmshToFoam.
The existing `FoamAgentExecutor.execute()` always calls `blockMesh`
as the first mesh-generation step — it has no awareness of an
already-existing polyMesh. For M7's eventual run path (Track B
end-to-end) to function, the executor must be able to skip
`blockMesh` when an upstream stage has already produced the polyMesh.

This DEC adds the smallest viable structural patch: a boolean flag +
override path on TaskSpec + a guard at the `blockMesh` call site +
geometry-dispatch skip. No `case_kind` dispatch change. No reading
of `case_manifest.yaml` from inside the executor. No new line-A →
trust-core data dependency.

## Pre-implementation surface scan (per DEC-V61-088)

Performed 2026-04-28 prior to code work:

1. **ROADMAP scan**: M6.1 maps cleanly to ROADMAP §M6.1 trust-core
   micro-PR per spec_v2 narrowing. No prior implementation matched.
2. **Existing-implementation grep**: searched repo for
   `mesh_already_provided`, `case_dir_override`, `blockMesh skip`,
   `polyMesh exists` — zero pre-existing hits in `src/`, `ui/`, or
   `tests/`. The blockMesh call at `src/foam_agent_adapter.py:696`
   was a single hardcoded `_docker_exec("blockMesh", ...)` line with
   no awareness of upstream-meshed cases. Surface clean for the new
   contract.

## Scope rationale — supersedes spec_v2 "flag only" narrowing

The spec_v2 M6.1 section narrowed M6.1 per Kogami M6 finding 4 to
"option-b · structural flag only, no case_kind dispatch." Codex
round-1 review (2 P2 findings) demonstrated the flag-only design is
non-functional: the boolean alone gives the executor no path to read
polyMesh from, and the geometry dispatch runs before any flag guard
could fire. This DEC therefore EXPANDS the narrowing to include
`case_dir_override` + dispatch-skip, justified by the round-1 P2
findings citation. The expansion stays inside the M6 finding-4
boundary spirit (no `case_kind` enum, no manifest-content reads from
trust-core) while making the contract actually usable by M7.

## Scope

### In scope (M6.1)

1. **TaskSpec fields** (`src/models.py`):
   - `mesh_already_provided: bool = False` — opt-in flag
   - `case_dir_override: Optional[str] = None` — caller-supplied path
     to a pre-populated case directory. Coupled with the flag: setting
     the flag without the override is rejected.

2. **Executor case-dir resolution** (`src/foam_agent_adapter.py:622+`):
   when the flag is set, use `case_dir_override` instead of allocating
   a fresh `ldc_{pid}_{ms}` temp dir under `self._work_dir`. The
   override path must exist and be a directory; otherwise fail fast.

3. **Executor dispatch skip** (`src/foam_agent_adapter.py:662+`): when
   the flag is set, skip the entire geometry-type dispatch (no
   `_generate_*` method runs). The case dir is expected to be
   pre-populated. Solver name routes to `simpleFoam` as a safe default
   that M7 can refine via its own dispatch logic.

4. **Executor blockMesh skip + defensive check**
   (`src/foam_agent_adapter.py:720+`): when the flag is set, skip
   `blockMesh` and verify `(case_host_dir / "constant" / "polyMesh")`
   exists. Fail fast otherwise (avoids running the solver on an empty
   mesh).

The first design surfaced a Codex round-1 P2 finding: a boolean alone
left the executor with no path to find polyMesh, and the geometry
dispatch ran before the guard, so the temp case got LDC scaffold even
when the flag was set. Adding `case_dir_override` + skipping the
dispatch closes both gaps. Scope grew from ~25 LOC main + ~40 LOC
tests to ~60 LOC main + ~120 LOC tests — still trust-core micro-PR
magnitude.

### Out of scope (deferred to M7 or later)

- New `case_kind` enum value (e.g., `imported_user`) in executor dispatch
- Reading `case_manifest.yaml` content from inside the executor
- Wiring `mesh_already_provided=True` from any caller (TaskSpec stays
  defaulted-False until M7 builds the dispatch path that sets it)
- snappyHexMesh execution (M7)
- Verdict ceiling logic for imported cases (M5.1)

## Why this is structurally safe (line-A vs trust-core boundary)

The `mesh_already_provided` flag is a **structural boolean** carried on
TaskSpec — it does not transport line-A-authored content into the
executor. The defensive `polyMesh.exists()` check is a pure filesystem
predicate. Neither reads `case_manifest.yaml`, neither reads any field
that line-A code writes. ADR-001 four-plane import direction is
preserved.

## Failure modes considered

| Failure mode | Mitigation |
|---|---|
| Flag accidentally set on a whitelist case → blockMesh skipped → solver runs on stale or wrong mesh | M6.1 sets the flag NOWHERE — only manual TaskSpec construction in M7 will set it. The dispatch wiring is M7's job, with its own gates. Until M7, flag is unreachable from production code paths. |
| Flag set + polyMesh missing → solver runs on empty mesh → nonsense residuals get reported as "valid" | Defensive `polyMesh.exists()` check fails fast BEFORE solver is invoked. |
| Whitelist case regression | Default-False flag; existing tests cover the default path. New regression test pins blockMesh-still-runs behavior on default TaskSpec. |
| Test-only path setting flag without polyMesh on disk | Test fixtures must explicitly pre-create the polyMesh dir; the defensive check makes this contract enforceable. |
