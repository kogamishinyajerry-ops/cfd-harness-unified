PR-5c.2 materially improves the M3 migration hazard: the upgrade note is accurate, the legacy-base64 probe is sane for a non-fatal migration hint, and the added sidecar edge-case tests cover the intended CRLF/BOM/whitespace paths. ### Medium (queue for follow-up)

   Evidence: `src/audit_package/sign.py:18-23`, `src/audit_package/sign.py:217-235`, `tests/test_audit_package/test_sign.py:317-338`, `pyproject.toml:35-37`

That leaves the source-only docstring as the main migration surface, which is weaker than the stated intent of M3 rec #2.
   Suggested fix: change the warning class to `FutureWarning`, `UserWarning`, or a custom `LegacyHmacSecretWarning(UserWarning)`; and mirror the migration note into a real operator doc surface such as `README.md` or the future verification/runbook doc referenced from `docs/ui_roadmap.md`.

## Per-area analysis
### 1. Migration note visibility
The note added at the top of `src/audit_package/sign.py` is technically correct and concise. It clearly states the behavioral change, gives the exact rewrite form (`base64:<same-value>`), and explains the divergence mechanism (`src/audit_package/sign.py:4-23`).

What it does **not** do is reach a realistic operator by itself. A module docstring is a source-reader surface, not an operational one. This repo's top-level `README.md` does not document the Phase 5 HMAC env-var contract today (`README.md:12-18`), and the roadmap still points toward a future verification doc / runbook surface (`docs/ui_roadmap.md:218-225`). So the upgrade note is good as a source artifact, but docs would be more effective in:

1. `README.md` once Phase 5 is exposed to normal setup/ops workflows.
2. The future verification procedure / runbook (`VERIFY.md`-style doc) that operators will actually follow.
3. Any startup/configuration doc for the UI backend signer service, once that wiring lands.

### 2. `_looks_like_legacy_base64` heuristic
The heuristic is acceptable for this use case. It requires all of:

- length `>= 24`
- standard base64 alphabet only
- length divisible by 4
- successful `b64decode(..., validate=True)`

That is narrow enough for a non-fatal migration hint (`src/audit_package/sign.py:117-148`). False positives remain possible on dense 24+ char plain-text secrets, but the consequence is only a warning, not a behavior change, and the `text:` escape hatch is sufficient (`src/audit_package/sign.py:229-231`).

The deliberate exclusions also make sense:

- URL-safe base64 does not warn.
- Unpadded strings do not warn.

Those cases are legacy-compat misses only if the project previously documented or relied on them, which it did not. For the stated target of catching old `openssl rand -base64 32` style secrets, the heuristic is appropriate.

### 3. Warning severity choice
`DeprecationWarning` is not the right class if the goal is "operators will see this at first signer startup." It is a reasonable class only if the audience is developers running tests with warning capture enabled. This repo does not enable global warning filters in pytest config (`pyproject.toml:35-37`), and the tests assert warning emission via `recwarn`, not real-world visibility (`tests/test_audit_package/test_sign.py:317-389`).

If the intent is an operator-directed migration message:

1. `UserWarning` is the clearest operational choice.
2. A custom subclass of `UserWarning` is even better if the team wants targeted filtering later.
3. `FutureWarning` is also better than `DeprecationWarning` for default visibility, but semantically it is still more "library consumer" than "operator action required".

### 4. Edge-case tests
The added tests cover the intended edges well:

- URL-safe base64 does not spuriously warn (`tests/test_audit_package/test_sign.py:374-383`)
- CRLF sidecars are accepted (`tests/test_audit_package/test_sign.py:468-473`)
- BOM-prefixed sidecars are rejected (`tests/test_audit_package/test_sign.py:475-480`)
- Trailing whitespace is stripped before validation (`tests/test_audit_package/test_sign.py:482-487`)

These are the right L1/L3-style edges to pin. I did not find a false negative in the current cases. They do not newly pin env-var prefix case-sensitivity or BOM-prefixed env vars, but that is a documentation/contract completeness gap, not a bug in this diff.

### 5. Anything new surfaced
Only the warning-visibility gap above. I did not find a new correctness, security, or test-contract bug in the changed files beyond that channel-choice issue.
