## R17 — verdict: CHANGES_REQUIRED · 1 P1 + 1 P3

**Finding P1 (HIGH)** — `bc_setup_from_stl_patches.py:518-527`
> "This scope reduction assumes broken `fvSolution`/`fvSchemes` overrides will be obvious at solve time, but the current solve paths do not actually surface the root cause inline. The blocking `/solve` route only returns `pimpleFoam exited with code …; see log...`, and the workbench's normal `/solve-stream` path turns `FOAM FATAL` into a non-fatal SSE `error` before ending in `phase="completed"`."

## R17 closure (commit `2ff11d9`)

- **P3 fix (verbatim)**: replaced two-stage line+block stripping with a single alternation regex (`/\*.*?\*/|//[^\n]*`) — eliminates the precedence bug R15 hit. New test `test_user_override_controldict_with_block_commented_icofoam_proceeds`.

Runtime divergence becomes machine-actionable without brittle pre-flight regex.

## R18 — verdict: CHANGES_REQUIRED · 2 P1

**Reviewed commit**: `2ff11d9` (R17 closure)

**Finding P1-A (HIGH)** — `solver_streamer.py:609-619`
> "When a streamed solve aborts with the `--> FOAM FATAL IO ERROR` variant (for example, missing dictionary keys), `fatal_seen` never flips because `_parse_line_to_events()` still only matches `^--> FOAM FATAL ERROR`. In that scenario the new terminal fields stay `failed=false` and `failed_reason=null` even though `_FOAM_FATAL_BLOCK` can parse the block, so a real OpenFOAM fatal still looks like an ordinary completion to the caller."

**Finding P1-B (HIGH)** — `solver_streamer.py:633-635`
> "For the runtime-fatal case this commit is trying to fix, adding `failed`/`failed_reason` here is not enough: `SolveStreamContext.tsx` still treats every `done` event as a successful completion, and `Step4SolveRun.tsx` calls `onStepComplete()` for any `phase === "completed"`. As a result, a FOAM-fatal streamed run still advances the workbench to Step 5 and renders the amber "completed" panel instead of an actionable rejection card."

## R18 closure (commit `7e9483a`)

- **P1-A (verbatim)**: extended `_FOAM_FATAL` line regex to alternate `ERROR|IO ERROR`. New backend test `test_done_event_carries_failed_reason_for_io_error_variant`.
- **P1-B (accept)**: wired backend's `failed` flag through to the frontend state machine.
  - `SolveStreamSummary` interface gains `failed?: boolean` + `failed_reason?: string | null`.
  - `SolveStreamContext` `done` event handler branches on `s.failed`: `true` → `setPhase("error")` + `setErrorMessage(failed_reason)`; `false` → existing `setPhase("completed")` + `removeQueries` (only on success — failed runs no longer drop the prior bundle).
  - `Step4SolveRun.tsx` unchanged: existing `phase === "error"` branch already maps to `onStepError` + rose-themed rejection panel, so the context-level routing change is the entire frontend surface area.
- 2 new frontend tests in `SolveStreamContext.failed.test.tsx` (failed=true → phase=error+failed_reason; failed=false → phase=completed, regression guard).
