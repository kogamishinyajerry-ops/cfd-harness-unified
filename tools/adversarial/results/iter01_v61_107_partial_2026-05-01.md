# DEC-V61-107 partial fix (2026-05-01) — fvSchemes upgrade + icoFoam stability finding

**Context**: DEC-V61-106 deferred iter01 migration to `analytical_comparator_pass` until icoFoam's structural divergence was fixed. The dt sweep (commit `cfb13f5`) ruled out CFL as the cause. This investigation chased deeper.

## Investigation chain

1. **Inspect authored 0/U + 0/p + system/fvSchemes + system/fvSolution + constant/polyMesh/boundary** for a fresh iter01 case_dir.
2. Found `divSchemes div(phi,U) Gauss linear` (central differencing — known unstable for convection-dominated flow) AND `laplacianSchemes default Gauss linear orthogonal` (assumes orthogonal mesh, but tetrahedral STL meshes are inherently non-orthogonal — mismatch with the `nNonOrthogonalCorrectors 2` already declared in fvSolution).
3. **Fix attempt 1**: change `divSchemes` → `Gauss linearUpwind grad(U)`. Test: still NaN (iter01 dt=1.0).
4. **Fix attempt 2**: also change `laplacianSchemes`/`snGradSchemes` `orthogonal` → `corrected`. Test: still NaN.
5. **Read the icoFoam log** at `<case_dir>/log.icoFoam` — found Courant numbers exploding step by step:
   - t=1s: CFL_max=226
   - t=2s: CFL_max=170,206
   - t=3s: CFL_max=388,794
   - t=4s: CFL_max=3,137,960
6. **Fix attempt 3**: enable `adjustTimeStep yes; maxCo 0.5;` in controlDict — reverted because **icoFoam ignores adjustTimeStep** (it's a pimpleFoam/transient-feature, not supported by icoFoam).

## What's shipped (kept)

`ui/backend/services/case_solve/bc_setup_from_stl_patches.py`:
- `divSchemes  { div(phi,U) Gauss linearUpwind grad(U); }` — second-order upwind (was central differencing)
- `laplacianSchemes { default Gauss linear corrected; }` — corrected for non-orthogonal mesh (was orthogonal)
- `snGradSchemes { default corrected; }` — same

These are universally-correct improvements regardless of iter01's eventual fix. LDC/channel pass cases unchanged (diffusion-dominated, scheme choice invisible). 4 PASS · 2 SKIP · 0 FAIL smoke baseline preserved. 15/15 BC mapper tests + 27/27 V61-106 tests pass.

## What's NOT shipped (deferred)

The actual iter01 fix needs one of:

**Option A — Switch to pimpleFoam**: pimpleFoam supports `adjustTimeStep yes; maxCo 0.5;`. Requires:
- New BC author template path (PISO → PIMPLE block in fvSolution)
- New controlDict (adjustTimeStep)
- Possibly new fvSchemes if pimpleFoam expects different defaults
- Tests for the new path
- Codex review

**Option B — Compute fixed dt from CFL**: at iter01's mesh resolution, dt < dx_min/U_max ≈ 0.0005/1.0 = 5e-4. Smoke would need 2,000+ steps to reach t=1s. Wall time would be unworkable for smoke.

**Option C — Coarsen the mesh** in the gap region to remove the small cells. Out of scope for BC mapper; would need a meshing-side tweak.

Recommended: **Option A** as DEC-V61-107.5 follow-up.

## Honest framing

Today's commit lands the universally-correct fvSchemes upgrade and confirms the iter01 root cause is icoFoam's lack of timestep adaption combined with high-aspect-ratio gap cells. iter01 stays SKIPPED in smoke pending the pimpleFoam migration. The diagnostic chain (dt sweep + log inspection) has narrowed the unknown from "iter01 diverges, why?" to "icoFoam can't adapt dt for this mesh; either swap to pimpleFoam or change the mesh".
